## 工作流程

### 第一步：确认要写哪篇文章

1. 读取 `go_agent_series_plan.md`，确认用户要求编写的具体文章
2. 确认文章所属章节（大模型基础/Agent认知/Eino入门/Eino进阶/RAG实战/项目实战/部署工程化/面试题库）

4. 检查 `src/go_agent_series/` 目录下是否已有该文件，避免重复创建

### 第二步：确定写作策略

根据文章所属章节，采用不同的写作策略：

**理论导向型**（大模型基础、Agent认知篇）：
- 注重概念的深入浅出讲解
- 理论要扎实，有体系感
- 大量使用类比、举例帮助理解
- 图文并茂是重点，多用架构图、流程图、对比图
- 适合作为小白入门的知识导论
- 理论占 60%，代码示例占 40%

**实战导向型**（Eino入门/进阶、RAG实战、项目实战、部署工程化）：
- 注重动手操作，步骤详细细致
- 逻辑清晰，前后关联紧密，不能有大的跳跃
- 知识点整体性强，不显得零碎
- 每个步骤都要说清楚应该怎么做
- 代码要完整、可运行、有注释
- 理论占 30%，代码实战占 70%
- 小白对照教程能一步一步跑通

**面试导向型**（面试题库）：
- 每道题要有深度分析和参考回答
- 分析部分像技术博文，深入浅出
- 回答部分像面试现场，简洁有力
- 覆盖高频考点，有实战感

### 长文章拆分策略

`go_agent_series_plan.md` 中规划的是一个大纲，实际写作时可能发现某篇文章的内容量远超一篇能容纳的范围。当遇到以下情况时，应主动将一篇拆分为多篇：

**判断标准**：
- 如果按规划写完，正文预计超过 10000 字（不含代码），考虑拆分
- 如果一篇文章覆盖的知识点之间相对独立，且每个知识点都值得深入展开，考虑拆分
- 如果代码示例有多个完整项目（不是递进式的小改动），考虑拆分

**拆分方式**：
- 文件名在原文件名基础上加后缀区分，比如 `eino_tools.md` 拆成 `eino_tools_function.md` 和 `eino_tools_builtin.md`，或者用 `_part1` / `_part2` 后缀
- 拆分后每篇文章仍然独立成文，有自己的 frontmatter、开篇引入和小结
- 拆分时要告知用户，由用户确认拆分方案后再动手写，同时提醒用户后续需要更新 sidebar 配置

**description 写作要求**：
- 长度控制在 80-120 字之间，这是搜索引擎摘要的最佳展示长度
- **必须优先突出高搜索量的热门关键词**，这些关键词要尽量靠前出现在 description 中。高搜索量关键词包括但不限于：`Go语言`、`Golang`、`大模型`、`LLM`、`AI Agent`、`RAG`、`ChatGPT`、`Prompt Engineering`、`Function Calling`、`向量数据库`、`Embedding`、`AI应用开发`、`智能体`、`大模型应用`、`AI编程`、`Eino`、`Go Agent教程`、`Agent教程`、`AI Agent教程`、`大模型教程`、`Go语言教程`、`入门教程`、`实战教程`。根据文章具体内容选择最相关的 3-5 个热门关键词自然融入，尤其注意融入"教程"类长尾关键词（如"Go Agent教程"、"AI Agent入门教程"），因为大量用户会搜索"XX教程"来寻找学习资料
- 关键词要自然融入句子，不能生硬堆砌，读起来像一句完整通顺的话
- 内容要概括本文的核心主题 + 读者能获得什么价值（学到什么、解决什么问题）
- 写法示例：`Go语言 AI Agent 开发实战：本文基于 Eino 框架，手把手教你用 Golang 构建多工具协作的大模型智能体应用，涵盖 Function Calling、工具调度策略与完整代码示例。`

#### 文章结构

1. **大标题**：`# **{标题}**`

2. **开篇引入**：1-2 段简洁的背景介绍，说明本文要讲什么、为什么重要。不需要回顾上一篇内容。

3. **正文章节**：标题必须带数字编号前缀，从二级标题开始：
   - 二级标题（##）：`## **1. 章节标题**`、`## **2. 章节标题**` 依次递增
   - 三级标题（###）：`### **2.1 子标题**`、`### **2.2 子标题**` 跟随父级二级标题编号
   - 四级标题（####）：`#### **2.1.1 子子标题**` 跟随父级三级标题编号

   每个章节围绕一个核心知识点展开。

4. **代码示例**：使用 ` ```go ` 包裹，代码要完整可运行，关键处有注释。代码后紧跟运行结果：
   ````
   ```go
   package main
   // 完整代码...
   ```
   运行结果：
   ```
   输出内容
   ```
   ````

   **代码组织策略**（根据代码量灵活选择）：

   - **单文件模式**：如果代码量不大（单个示例不超过 300 行），一个 `main.go` 就够了，不需要拆分，保持简洁即可
   - **多文件模式**：当代码量较大（单个示例超过 300 行）或者 逻辑较复杂（比如涉及多个模块交互、自定义类型较多）时，可以拆分成多个 Go 文件。拆分时必须做到：
     1. 在代码示例之前，先用文字说明整个项目的文件结构，例如：
        ```
        项目结构：
        my_agent/
        ├── main.go          // 程序入口
        ├── agent.go         // Agent 定义与配置
        ├── tools.go         // 工具函数定义
        └── go.mod
        ```
     2. 每个文件的代码都要完整展示，文件名用注释或标题标明（如 `// agent.go` 或用三级标题 `### agent.go`）
     3. 最后要给出完整的运行命令，让读者能一步到位地跑起来

   **代码解释原则**：

   - 如果代码逻辑简单、注释已经足够清晰，不需要额外解释，避免啰嗦
   - 如果代码中有初学者不容易理解的部分（比如 channel 的用法、context 传递、接口实现的设计意图等），在代码块后面用一小段文字重点解释这些难点即可，不需要逐行翻译代码

   **外部依赖说明要求**（实战篇重点）：

   当代码示例引入了**本篇文章中首次出现的外部依赖**，必须在首次使用时提供完整的安装/准备说明（如果同一篇文章后面的示例再次用到同一个依赖，就不需要重复写安装步骤了）：

   - **Go 库依赖**：给出 `go get` 命令
   - **外部服务依赖**（如向量数据库 Milvus/Weaviate、消息队列、Redis 等）：给出安装方式，优先推荐 Docker 一键启动的方式（因为最简单），并附上验证服务是否正常运行的方法
   - **API Key / 账号**：如果需要注册第三方服务获取 Key，说明在哪里注册、怎么获取
   - **环境变量**：列出需要设置的环境变量及其含义

   示例格式：
   ````
   ### 环境准备

   本节示例需要用到 Milvus 向量数据库，我们用 Docker 快速启动：

   ```bash
   go get github.com/milvus-io/milvus-sdk-go/v2

   ````

### 第四步：插入图片占位标记

在文章中需要图片辅助理解的位置，插入图片占位标记。图片是让教程"活起来"的关键，不能过少。

**理论型文章**：至少 5-8 张图（架构图、流程图、对比图、概念示意图等）
**实战型文章**：至少 3-5 张图（架构图、流程图、效果截图占位等）

**图片占位标记格式**：

```
> 【建议配图{序号} —— {图片标题}】
>
> 图片描述：{详细描述图片内容，包括布局、元素、颜色、箭头、图标等，要详细到设计师看了就能画出来}
>
> 整体目的：{这张图要帮助读者理解什么}
```

**图片设计原则（极其重要，务必严格遵循）**：

图片设计是一种创作，不是模板匹配。**核心目标只有一个：让读者看到图的一瞬间就理解文字想表达的意思，甚至比读文字更快更直观。** 每张图都要做到生动形象、视觉上好看、结构丰富多样、能准确表达文字意思并辅助理解。

全局 skills 目录下 `llm-interview/references/` 中的两份 PDF 文件是图片设计的黄金标杆。每次设计图片时，必须先读取这两份 PDF，把它们中的配图水准作为创作目标，然后根据当前文字内容自由创作。不要去提取 PDF 中的图属于什么"类型"然后对号入座——而是去感受那些图为什么好看、为什么一眼就能看懂，然后在你自己的创作中达到同样的效果。

好的图片描述必须做到以下几点：

1. **结构形式丰富多样，完全由内容驱动**。每张图的视觉结构应该完全由它要表达的文字内容来决定，像设计师一样根据具体内容自由创作最合适的视觉形式。一篇文章中的多张图，尽量让整篇文章的配图看起来丰富、有变化、不单调。绝不能所有图都是同一种结构（比如全是左右对比，或全是线性流程）。视觉形式没有任何限制——可以是任何你认为最能表达这段文字内容的结构，创意越丰富越好。**特别注意：避免过度使用环形图、同心圆、洋葱圈结构**——这类结构容易变成万能模板被滥用，只有当内容确实是"循环流程"或"迭代闭环"时才适合用环形。

2. **用图标和视觉隐喻替代纯文字方框**。不要只写"画一个方框写上'用户输入'"——要描述具体的图标：用户头像、齿轮、放大镜、警告三角、机器人、文档、数据库圆柱、大脑、锁、云等等。每个概念都应该有对应的视觉符号，让图在缩小到很小时仍然能通过图标识别含义。

3. **颜色要有语义，不只是区分**。红色=错误/风险/问题，绿色=正确/通过，蓝色=正常流程，橙色/黄色=警告/需注意，灰色=被淘汰/不推荐。让读者不看文字只看颜色就能感知信息的情绪和重要程度。

4. **箭头和连接线要讲故事**。粗箭头=主流程，虚线箭头=可选/回退/映射，红色箭头=错误传播，带标注的箭头=解释因果关系，双向箭头=双向交互。箭头不只是连接，它要表达关系。

5. **层次感和视觉重点**。图中要有明确的视觉焦点——最核心的元素最大、最醒目，辅助元素小且颜色淡。不要所有元素一样大一样重，读者应该第一眼就知道先看哪里。

6. **标注文字要精炼**。图中的文字是高度凝练的短语或关键词，每个标注控制在 10 个字以内。超过 15 个字的信息应该放在��文而不是图里。

7. **白色/浅色干净背景**。不要使用网格背景、深色背景或花哨的纹理。图片背景保持白色或非常浅的纯色，让信息元素本身成为视觉焦点。参考 PDF 标杆中的图片，全部都是干净的白底。

描述图片时，像一个设计师在构思作品：先想清楚这张图要帮读者理解什么（目的），然后自由构思最能达成这个目的的视觉形式（创作），再详细描述整体布局、各区域的具体内容（图标、颜色、标注文字）、元素之间的连接关系。描述要具体到"设计师看了就能画出一张好看且信息清晰的图"。

## 写作风格要求

1. **通俗易懂**：像一个有经验的前辈在跟你聊天，避免生硬术语堆砌。复杂概念用类比、举例来解释。

2. **以文字叙述为主，切忌大量列点**：这是最重要的风格要求。文章要像一个技术博主在写给人看的深度文章，有思路、有逻辑、有论据、有细节，读起来像是一个懂行的人在跟你聊天。**大量列点罗列要点和总结是最典型的 AI 写作痕迹，必须避免。** 具体要求：
   - 讲解概念、分析优劣势时，用连贯的段落文字来表达。但并不是说把所有东西杂糅到一段里——如果一个话题有多个方面（比如优点和缺点），完全可以用两段或多段来分别展开，每段聚焦一个方面，这样既有段落感又不凌乱
   - 有时候该分点就分点，比如分步骤操作（"第一步…第二步…"）或技术清单（表格），这些场景用列表是合理的。关键是这种形式在一篇文章中不能过多，文章整体必须以段落叙述为主
   - **小结部分的写法极其重要**：小结不是"从头到尾复述我们干了啥"的总结题，而是作者以自然的笔触收束全文。要做到首尾呼应，用一两段话把文章的核心思想提炼升华，起到画龙点睛的作用。写法上就当作文章还没写完，你需要用最后一两段话完成它，行文自然、内容连贯，读起来是文章的有机组成部分而不是一个独立的总结清单
   - 参考 `src/go_series/go_advanced/channel.md` 的写法——它用的就是段落叙述 + 代码示例的方式

3. **代码驱动**：每个知识点尽量配完整可运行的 Go 代码示例，包含代码执行结果。代码示例要递进式，从简单到复杂。

4. **逻辑连贯**：文章内知识点之间有清晰的递进关系，像讲故事一样，一个知识点自然引出下一个。

5. **注重细节**：讲解常见错误和踩坑点，提供最佳实践建议。

6. **全中文表达**：技术术语保留英文（如 Agent、RAG、Embedding、Token），解释和论述全部用中文。

7. **适度幽默**：延续现有系列风趣幽默的风格，但不要过度，保持专业感。

8. **篇幅控制**：每篇文章正文控制在 5000-10000 字之间（不含代码），代码示例根据需要可以更长。

9. **标题规则**：标题不要用类似”多轮对话：让模型记住上下文”这种格冒号分离，前面标题，后面解释的格式，一看就是AI写的，像这样的标题直接用简短的描述“多轮对话”就可以了，标题一定要简明扼要，也不要出现类似“闭源模型（Proprietary Models）这种标题是中文然后打个括号英文的格式，一看就是AI写的，直接用前面的中文标题节课，括号英文解释不需要，如果是一些专业名词，比如Agent，Rag之类的话，那标题直接用英文即可，总之，标题不需要中英文混杂，解释

10. **减少AI味儿**：行文一定要减少AI味儿，要让人看不出来是AI写的，比如用词上，像“朴素”这类词就不应该出现在技术教程里，应该改成“基础”这个词来描述

## 关键原则

1. **每篇文章都要先读取规划文档** `go_agent_series_plan.md`，确认内容要点不遗漏。
2. **先读取一篇现有文章**（如 `src/go_series/go_advanced/channel.md`），确认格式一致。
3. **代码必须完整可运行**，不要只给片段。关键依赖要说明安装方式。
4. **图片占位标记不能少**，理论型文章至少 5 张，实战型文章至少 3 张。
5. **不需要前后衔接**：开头不回顾上篇，结尾不预告下篇。每篇文章独立成文。
6. **结尾推广块必须加**，使用上面提供的统一 HTML 模板。
7. **涉及 Eino 框架的文章**，代码要基于字节跳动 CloudWeGo Eino 框架（`github.com/cloudwego/eino` 和 `github.com/cloudwego/eino-ext`），确保 API 用法准确。如有不确定的 API，先通过 context7 查阅 Eino 最新官方文档（cloudwego.io）。
8. **实战篇的代码示例必须让读者能跑起来**：如果涉及外部依赖（Go 库、数据库、第三方服务等），必须在代码前给出完整的安装/启动步骤。如果代码涉及多个文件，必须说明项目结构和运行命令。面向初学者，不能假设读者知道怎么装某个数据库或怎么组织多文件项目。
9. **长文章主动拆分**：如果按规划文档写某篇文章时发现内容量过大（正文超过 10000 字），应主动提出拆分方案，经用户确认后再写。拆分后的文章各自独立成文。
10. **大模型 API 调用统一使用通义千问（DashScope）的 OpenAI 兼容接口**，不使用 Google Gemini 等国外大模型 API（因为作者是中国用户，无法付费使用）。具体要求：

   - API Base URL：`https://dashscope.aliyuncs.com/compatible-mode/v1`
   - API Key 环境变量：`DASHSCOPE_API_KEY`
   - 默认模型：`qwen-plus`（通用场景），`qwen-turbo`（轻量场景），`qwen-max`（复杂推理场景）
   - 首次出现代码示例时，需要说明安装依赖和申请 API Key 的方式（阿里云百炼平台）
