代码跑不起来、报了一串看不懂的错、功能行为不对——调试是写代码绕不开的一环。好消息是，调试恰恰是 AI 特别能帮上忙的地方：它读得懂报错、能顺着代码追根因、能给出修复方案。但前提是你得会喂——同样一个 bug，有人三言两语让 AI 一击命中，有人折腾半天它还在瞎猜，区别全在于你给了它什么信息。

这一篇讲透 AI 辅助调试：怎么把报错正确地喂给 AI、什么信息组合最有效、常见 bug 怎么调，以及那个 Vibe Coding 特有的方法论——让 AI 修它自己写出的 bug。

## **1. 调试的关键是喂对信息**

AI 调试调不准，几乎都是因为你给的信息不够。它看不到你的屏幕、不知道你刚才操作了什么、不清楚完整的报错，只能基于你那句模糊的「报错了帮我看看」去猜。

所以调试的第一原则是：**把信息喂全、喂准。** 一次有效的调试求助，至少要包含三样东西——完整的报错信息（不是截一半、不是转述，是原文）、复现步骤（你做了什么导致它出错）、相关的代码（出错附近的代码，用 `@` 引用最好）。把这三样给齐，AI 往往一次就能定位；缺了任何一样，它就得靠猜，来回好几轮。

## **2. 报错信息怎么喂**

报错信息是调试最关键的线索，喂的时候有几个讲究。

**给完整的、原始的报错，别转述、别截断。** 很多人只贴报错的最后一行，但真正有用的信息常在调用栈（stack trace）里——它告诉你错误是从哪一层、哪个文件、哪一行传上来的。把完整的报错连同调用栈一起贴给 AI，它能顺着这条链定位到根源。如果是终端里的报错，直接复制全部文字；如果是浏览器，打开控制台把红色报错完整复制。

**报错配上对应的代码。** 光有报错，AI 还得猜对应代码长什么样。用 `@` 把报错涉及的文件引用进来，或把出错附近的代码贴上，让它对着真实代码分析，而不是凭报错想象。

**Prompt：**
```
我的代码报了这个错（完整报错和调用栈）：
{粘贴完整报错}

相关代码在这里：@{出错的文件}
复现步骤：{我做了什么操作触发的}

请帮我定位根本原因，并给出修复方案。先说清楚为什么会错，再给改法。
```

注意最后一句「先说清楚为什么会错」——这能避免 AI 不分析原因就瞎改。让它先讲清根因，你既能判断它是否真找对了，也能学到这个 bug 是怎么回事。

## **3. 组合更多信息**

有些问题光靠报错说不清，尤其是界面行为不对、但不报错的情况。这时要组合更多信息。

**界面问题用截图。** 页面布局错乱、按钮点了没反应、显示和预期不符——这些用文字很难描述清楚，直接截图给 AI（Cursor、Claude Code 都支持发图）。它能直接看到现象，比你描述「那个按钮歪了」准得多。

**行为问题加日志。** 如果一个功能行为不对但不报错，让 AI 帮你在关键位置加上日志（打印出中间变量的值），跑一遍，把日志输出再喂给它。这样它能看到数据在哪一步变得不对，顺藤摸瓜找到问题。这是排查「没报错但结果不对」类 bug 的利器。

**把现象和预期都说清。** 描述 bug 时，既说「实际发生了什么」，也说「我期望它怎样」。两者的差距正是问题所在，AI 看到差距才知道往哪查。

## **4. 让 AI 修它自己写的 Bug**

Vibe Coding 有个特有的循环：AI 写的代码出了 bug，让 AI 自己修。这听起来理所当然，但用好它有方法。

最直接的，是把 bug 反馈回去让它修——但同样要喂全信息，别只说「你写的有问题」。把报错、现象、它刚写的代码一起给，让它分析自己哪里错了。AI 修自己的代码通常比修陌生代码更顺，因为相关上下文还在。

要警惕的是「越修越乱」：有时 AI 修一个 bug 会引入新的，或者反复在两个错误方案之间打转。一旦发现它修了几轮还没好、甚至越改越糟，就别让它继续硬修了——停下来，换个思路：让它先彻底解释这段代码的逻辑和这个 bug 的根因（用 Ask 模式、只分析不改），你和它一起把问题想清楚，再让它针对根因改。必要时用 `/rewind` 或 git 把它改乱的退回去，从一个干净的状态重新来。

还有个保险：让 AI 修复时一并补上能复现这个 bug 的测试。这样修完跑测试就能确认真修好了，将来也不会再犯（回归）。

## **5. 一个完整的调试实战**

把方法串起来，走一遍真实的调试流程。假设你做的待办应用，点「删除」按钮后页面没反应，控制台报了个错。

第一步，**收集信息**：打开浏览器控制台，把红色报错连同调用栈完整复制；记下复现步骤（点某个待办的删除按钮就报错）；找到删除相关的代码文件。

第二步，**喂给 AI 定位**：用上面第 2 节的 Prompt 模板，把完整报错、复现步骤、`@` 引用的相关代码一起发给它，让它先说根因再给改法。它分析后告诉你：删除函数里用了一个 undefined 的 id，因为按钮传参时漏了。

第三步，**验证并加保险**：让它修复，并补一个测试覆盖「删除某条待办」。修完你亲自点一遍删除按钮确认好了，跑测试也绿了。

第四步，**举一反三**：问它一句「类似的 undefined 传参问题，项目里别处还可能有吗？」让它顺手排查同类隐患。

这套流程的核心，始终是「喂全信息 → 让它先析因再改 → 验证 → 防回归」。bug 大小不同，流程一样适用。

## **6. 常见误区**

**只说「报错了」不给细节。** AI 不是读心术，信息不全它只能猜。完整报错、复现步骤、相关代码，一样别少。

**让 AI 不分析就改。** 不让它说根因，它可能瞎改、改错。坚持「先析因、再动手」。

**陷入越修越乱还硬修。** 修几轮不好就止损，回退到干净状态、先想清根因再来，别在错误的方向上越陷越深。

**修完不验证、不防回归。** 修了要亲自验、补测试，否则可能没真修好、或将来又犯。

## **7. 小结**

AI 是极强的调试搭档，但它的能力上限由你喂的信息决定。一次高效的调试，靠的是把完整报错、复现步骤、相关代码喂全喂准；界面问题配截图、行为问题加日志、说清现象与预期的差距。让它先分析根因再动手，修完验证并补回归测试。

而 Vibe Coding 特有的「让 AI 修自己写的 bug」要用得清醒：它修自己的代码通常更顺，但一旦越修越乱，就果断止损、回退、用 Ask 先把根因想透再改。掌握了这套「喂对信息、先析因、会止损、防回归」的章法，再头疼的 bug，你和 AI 也能稳稳地拆解掉。
