很多人对 Vibe Coding 有个误解：以为就是抛一句「帮我做个待办应用」，然后等 AI 把一切搞定。小项目这么干也许能蒙混过去，但凡是稍微复杂一点的项目，这样做几乎必然翻车——AI 会基于一个模糊的指令自由发挥，做出来的东西结构混乱、方向跑偏，等你发现，一堆代码已经缠在一起难以收拾。

恰恰相反，AI 时代比以往更需要规划。因为 AI 写代码飞快，一旦方向错了，它错得也飞快、错得也更多。规划就是在动手之前把方向定准，让 AI 这台高速引擎朝着对的方向使劲。这一篇讲透 Vibe Coding 场景下怎么做项目规划：先写什么、怎么用 AI 帮你选型和设计结构、规划文档怎么写、又怎么把规划交给 AI 执行。

## **1. 为什么 AI 时代更要规划**

传统开发里，写代码慢，所以你常常边写边想、慢慢调整方向。AI 时代变了：让 AI 生成代码只要几秒，瓶颈不再是「写得快不快」，而是「方向对不对」。方向对，AI 帮你飞；方向错，AI 帮你把错误的东西飞快地堆成山。

所以规划的价值被放大了。在动手前花十分钟把要做什么、怎么组织想清楚，能省下后面几小时甚至几天的返工。这不是回到繁琐的传统流程，而是抓住最高杠杆的那个动作——**先想清楚，再让 AI 放手做。** 你负责定义问题和方向，AI 负责高效实现，这正是人机协作里最该由你把住的部分。

## **2. 先写 PRD 再写代码**

规划的第一步，是把「要做什么」写清楚，也就是一份简单的 PRD（产品需求文档）。别被这个名字吓到，它不必正式——核心就是把你脑子里模糊的想法，落成一段清晰的文字：这个项目是干什么的、要解决什么问题、有哪些核心功能、每个功能大概什么样、给谁用。

写 PRD 这件事本身也能让 AI 帮你。你把模糊的想法丢给它，让它帮你理清、追问、补全：

**Prompt：**
```
我想做一个帮自由职业者管理项目和发票的小工具。请帮我把它整理成一份简单的 PRD：
1. 一句话定位
2. 目标用户和他们的核心痛点
3. MVP（最小可用版本）必须有的 3-5 个核心功能，每个一句话说清
4. 暂时不做、留到以后的功能
针对不清楚的地方，先反问我几个关键问题，再给文档。
```

这个 Prompt 的关键在最后一句——让它先反问。模糊的需求里藏着很多你自己都没想清的细节，让 AI 先问几个关键问题，逼你把想法想透，比它直接瞎猜强得多。来回一两轮，一份清晰的 PRD 就成型了，它将成为后面所有开发的方向锚点。

## **3. 用 AI 辅助技术选型**

想清楚做什么，接下来是「用什么做」。技术选型对新手是个难关——框架、数据库、部署平台一大堆，不知道怎么选。这正是 AI 的强项：它见过海量项目，能基于你的需求和水平给出建议。

**Prompt：**
```
我要做上面这个自由职业者管理工具，我是编程新手、想快速做出来并部署上线。
请帮我做技术选型，给出推荐方案并说明理由：
- 前端用什么、后端用什么、数据库用什么、部署用什么
- 优先考虑：上手简单、生态成熟、能免费部署
- 每一项给出推荐和一个备选，简要说明各自利弊
```

注意这个 Prompt 把你的约束说清了——新手、要快、要能免费部署。约束给得越明确，它的建议越贴合你的实际，而不是甩给你一套适合资深团队的复杂技术栈。拿到建议后，对不确定的点继续追问「为什么选这个不选那个」，既做了选型，又顺便学到了取舍的道理。

## **4. README 驱动开发**

一个很实用的规划技巧叫 README 驱动开发：在写任何代码之前，先把项目的 README 写出来——就好像项目已经做完了一样。在 README 里写清楚这个项目是什么、怎么安装、怎么使用、有哪些功能、API 长什么样。

这么做的妙处是，写 README 的过程会逼你把项目的样子在脑子里完整跑一遍：用户怎么用、接口怎么设计、功能怎么组织。很多设计上的问题，会在你写「使用说明」时就暴露出来——如果一个功能你连怎么用都说不清，那它多半还没设计好。等 README 写顺了，你对要做的东西就有了清晰的图像，再让 AI 照着实现，方向就稳了。

>
> **生图提示词（可直接发给 ChatGPT / 文生图工具）：**
> 画一张干净白底、现代扁平风格的「倒序」示意图。常规认知是「先写代码后写文档」，这里反过来。左边画一份 README 文档（图标含「项目是什么/怎么用/功能/API」几个小标题），标「① 先写 README，像项目已完成那样」，旁边一个灯泡图标标「写的过程暴露设计问题」。中间一个加粗箭头。右边画 AI（机器人）照着 README 产出整齐的代码和成品，标「② 再让 AI 照着实现」。底部小字「连怎么用都说不清的功能，说明还没设计好」。配色语义：绿=先想清楚的文档、蓝=照着实现。图片右下角放置引流信息：公众号：IT杨秀才 ｜ https://golangstar.cn（小字、浅灰、不抢主体）。

> 整体目的：让读者理解 README 驱动开发——先把使用说明写出来逼自己想清设计，再让 AI 照着实现。

## **5. 项目结构与模块拆分**

方向和技术定了，再想想代码怎么组织。好的项目结构让 AI 和你都不容易迷路；混乱的结构会让 AI 越改越乱、文件越堆越散。

模块拆分有个朴素原则：**按职责分开，一个模块管一件事。** 比如一个 Web 应用，把处理请求的、处理业务逻辑的、访问数据的分到不同地方，而不是全揉在一个文件里。这样改一处不至于牵连一片，AI 定位和修改也更准。结构怎么设计，同样可以让 AI 给建议——把你的技术栈和功能告诉它，让它推荐一个清晰的目录结构。

**Prompt：**
```
我用 Next.js + PostgreSQL 做这个项目，核心功能是项目管理和发票管理。
请推荐一个清晰、好维护的目录结构，说明每个目录放什么。
原则：按职责分层、新手能看懂、便于后续扩展。
```

把它给出的结构和你的 PRD 对一遍，确认每个功能都有地方放，这个结构就能当作开发的骨架。一开始结构不必复杂，够当前需要就好，随着项目长大再调整。

> 【建议配图3 —— 按职责分层的项目结构】
>
> **生图提示词（可直接发给 ChatGPT / 文生图工具）：**
> 画一张干净白底、现代扁平风格的分层结构图。纵向画几层，每层一个职责、一个图标和颜色：顶层「界面/页面层」（浏览器图标，蓝）「展示与交互」；中层「业务逻辑层」（齿轮图标，绿）「核心规则」；下层「数据访问层」（数据库圆柱图标，橙）「读写数据」。层与层之间用箭头表示调用方向（上层调下层）。右侧对照画一个反例小图：所有东西揉成一个乱糟糟的大文件（红叉，标「全揉一起，改一处牵连一片」）。配色语义：分层用蓝绿橙清晰区分、红=反例混乱。图片右下角放置引流信息：公众号：IT杨秀才 ｜ https://golangstar.cn（小字、浅灰、不抢主体）。
>
> 整体目的：让读者理解按职责分层的结构（界面/业务/数据）比全揉一起更清晰、更好让 AI 维护。

## **6. 规划文档模板**

把前面几步沉淀成一份规划文档，放进项目（比如叫 `PLAN.md` 或写进 `AGENTS.md` / `CLAUDE.md`），它既是你的蓝图，也是喂给 AI 的方向。一份实用的规划文档模板大致是这样的公式：

```markdown
# 项目规划：{项目名}

## 一句话定位
{这个项目是什么、给谁用、解决什么问题}

## 技术栈
- 前端：{...}  后端：{...}  数据库：{...}  部署：{...}

## 目录结构
{贴上按职责分层的目录结构，注明每个目录放什么}

## MVP 功能清单（按优先级）
1. {功能一：一句话描述 + 验收标准}
2. {功能二：...}
3. {...}

## 暂不实现
{留到以后的功能，避免 AI 跑偏去做}

## 约定
{编码规范、命名约定、不要做的事，如"不引入新依赖先问我"}
```

这份文档的价值，是让你和 AI 始终对着同一张蓝图干活。AI 每次接活前读它，就知道项目的全貌和边界，不会自由发挥跑偏。

## **7. 把规划交给 AI 执行**

有了规划文档，执行就顺了。一个稳妥的做法是：让 AI 照着规划，一个功能一个功能地实现，而不是让它一口气把整个项目做完。

复杂功能先让它用 Plan 模式出方案（前面 Claude Code、Cursor 篇都讲过），对照你的规划文档审一遍方向，再让它动手。每做完一个功能就验证、就提交，确认没问题再做下一个。这种「按规划、分功能、小步推进」的节奏，配合前面立好的规划文档，能让一个稍复杂的项目稳稳地从蓝图变成现实，而不是在一团乱麻里挣扎。

> 【建议配图4 —— 从规划到执行的闭环】
>
> **生图提示词（可直接发给 ChatGPT / 文生图工具）：**
> 画一张干净白底、现代扁平风格的流程闭环图。①「规划文档（蓝图）」（文档图标，绿）；箭头到②「按优先级取一个功能」（清单勾选图标，蓝）；到③「复杂的先 Plan 出方案、对照蓝图审」（蓝图+放大镜，紫）；到④「AI 实现这个功能」（机器人+代码，橙）；到⑤「验证+提交」（对勾+git 图标，绿）；再用一条回流箭头回到②「取下一个功能」，形成循环。中心标「按规划、分功能、小步推进」。配色语义：绿=蓝图与验证、蓝=取活、紫=审方向、橙=实现。图片右下角放置引流信息：公众号：IT杨秀才 ｜ https://golangstar.cn（小字、浅灰、不抢主体）。
>
> 整体目的：让读者掌握把规划落地的节奏——对着蓝图、一个功能一个功能地 Plan、实现、验证、提交，循环推进。

## **8. 常见误区**

**一上来就让 AI 做整个项目。** 项目越复杂越不能这样，它会跑偏、会缠成一团。先规划、再分功能小步做。

**规划做得太重。** 也别走另一个极端，写一份几十页的正式文档。Vibe Coding 的规划要轻——一份说清定位、技术栈、结构、功能清单的文档就够，重点是想清楚，不是写得正式。

**规划完就扔。** 规划文档是活的，随项目演进要更新。需求变了、结构调了，同步改文档，让它始终是 AI 和你共同的最新蓝图。

## **9. 小结**

AI 写代码越快，规划就越重要——因为方向错了，AI 只会更快地把错误堆成山。Vibe Coding 的规划不复杂：先用一份轻量 PRD 把「做什么」想清楚（让 AI 帮你追问理清），用 AI 辅助把技术栈和结构定下来，用 README 驱动逼自己想透设计，再沉淀成一份说清定位、技术栈、结构、功能清单的规划文档。

有了这张蓝图，执行就是对着它、一个功能一个功能地让 AI 实现、验证、推进。你负责想清楚方向，AI 负责高效落地——这正是 Vibe Coding 里最该由你把住的环节。把规划这一步做扎实，再复杂的项目也能稳稳地从想法变成现实。
