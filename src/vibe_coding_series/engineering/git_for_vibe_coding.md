如果说有一项技能是 Vibe Coding 时代绝不能跳过的，那就是 Git。很多新手觉得 Git 麻烦、可有可无，但当你让 AI 大刀阔斧地改代码时，Git 就从「专业工具」变成了「保命的安全绳」——它让你敢放手让 AI 改，因为无论改成什么样，你都能退回到之前的任何一个版本。

这一篇讲透 Vibe Coding 场景下的 Git：为什么它在 AI 时代尤其重要、最少要会哪几个操作、怎么用分支保护主代码、怎么让 AI 帮你写提交信息，以及最关键的——AI 把代码改坏了，怎么干净利落地退回去。

## **1. 为什么 AI 时代更需要 Git**

传统开发里，代码是你一行行写的，每一处改动你都心里有数。AI 时代不一样：你让 AI 改一个功能，它可能一口气动了五六个文件、改了上百行，速度快到你来不及细看。这种「大批量、高速度」的改动，正是 Git 大显身手的地方。

Git 给你三样东西，每一样都是放手用 AI 的前提。**它是后悔药**——AI 改坏了，一条命令退回改之前的样子，损失为零。**它是记录仪**——每一次提交都是一个存档点，你能清楚看到每一步改了什么、随时回到任意一个节点。**它是隔离带**——用分支，你可以让 AI 在一个独立的分支上折腾，不影响那份能正常跑的主代码。没有 Git，你让 AI 大改时是在走钢丝；有了 Git，你是系着安全绳在跳。

## **2. Git 基础操作速成**

不必精通 Git，会几个核心操作就够日常用。这几条是底座：

```bash
git init                  # 在项目里初始化 Git（一个项目只需一次）
git status                # 看当前有哪些改动
git add .                 # 把所有改动加入暂存区（准备提交）
git commit -m "提交说明"   # 把暂存的改动存成一个版本（存档点）
git log --oneline         # 看提交历史（每个存档点）
```

工作流就是一个循环：改代码（或让 AI 改）→ `git status` 看改了什么 → `git add` 暂存 → `git commit` 存档。每完成一个小功能、修好一个 bug，就提交一次，给项目留一个干净的存档点。提交得越勤、每次提交越聚焦一件事，将来回退和排查就越方便。

如果你的代码要备份到云端、或多人协作，再加上和远程仓库（如 GitHub）相关的几个：`git clone`（克隆远程仓库）、`git push`（推到远程）、`git pull`（拉取远程更新）。这些 AI 也能帮你——记不住命令时，直接问它「我要把当前改动推到远程 main 分支，命令是什么」，它给你现成的。

## **3. 分支策略与主分支保护**

分支是 Git 给 Vibe Coding 的一大利器。核心思想是：**保护好那份能正常跑的主代码，让 AI 在分支上折腾。**

最简单实用的策略是：主分支（main）始终保持可用、稳定；要做新功能或让 AI 大改，先开一个新分支，在分支上让 AI 改、测试、调好，确认没问题再合并回主分支。这样即使 AI 在分支上把代码改得一团糟，你的主分支始终是干净可跑的，大不了把这个分支删掉重来。

```bash
git checkout -b feature/login   # 新建并切换到一个功能分支
# 在这个分支上让 AI 干活、提交……
git checkout main               # 切回主分支
git merge feature/login         # 确认没问题后合并回来
```

对一个人开发的小项目，这套也许显得正式，但只要你打算让 AI 做有点风险的大改动，「开个分支试」永远是稳妥的。它把「试验」和「稳定版」隔开，让你能大胆地让 AI 尝试激进方案，反正主分支兜底。

## **4. 用 AI 写 Commit Message**

提交信息（commit message）写得好，将来回看历史时一目了然；写得差（一堆「修改」「更新」），等于没写。而写提交信息恰恰是 AI 的拿手活——它能看着你的改动，自动生成一条规范、清晰的提交信息。

最直接的方式，是把改动给它让它写：

**Prompt：**
```
看一下我当前 git 暂存区的改动（git diff --staged 的内容如下）：
{粘贴 diff，或在支持的工具里它能自己读取}

生成一条符合 Conventional Commits 规范的中文提交信息：
- 用 feat/fix/docs/refactor 等前缀
- 标题简洁、不超过 50 字
- 改动较多则正文分点列出关键变更
```

更省事的是把它做成一个命令或技能——前面 Claude Code 的命令篇、Codex 的命令篇都演示过 `/commit` 这样的自定义命令，它能自动读取改动、生成提交信息。配好之后，每次提交敲一下就行，团队的提交风格也统一了。规范的提交历史，是项目可维护性的一部分，让 AI 帮你把这件小事做好，长期收益不小。

## **5. AI 改坏了怎么回退**

这是 Git 在 Vibe Coding 里最关键的用途——AI 把代码改坏了，怎么干净地退回去。分几种情况。

**改动还没提交，想全部丢弃**，回到上一个提交的干净状态：

```bash
git checkout .        # 丢弃所有未暂存的改动
git reset --hard HEAD # 彻底回到最近一次提交的状态（慎用，会丢掉未提交改动）
```

**已经提交了，但想退回到之前某个版本**：先 `git log --oneline` 找到那个好的提交的编号，再回退过去。如果改动还没推到远程，`git reset` 能把分支指回去；如果已经推送或想保留历史，用 `git revert` 生成一个反向提交更安全。具体命令记不住没关系——把情况描述给 AI，让它给你确切的命令：「我刚才让 AI 改的几个提交把功能弄坏了，想退回到 abc123 这个提交，改动还没推送，该用什么命令」。

除了 Git，前面工具篇还讲过各工具自带的回退：Claude Code 和 Cursor 的 checkpoints / `/rewind`，能在两次 git 提交之间的更细粒度上回退 AI 的单步改动。两者配合：细粒度的单步回退用工具的 checkpoint，整段的版本回退用 Git。有了这两层保险，你完全不必怕 AI 改坏——坏了就退，退到干净状态重来。

## **6. 常见误区**

**不用 Git 就让 AI 大改。** 等于没安全绳走钢丝。哪怕个人小项目，让 AI 动大刀子前先 `git init`、先提交。

**几天才提交一次、攒一大堆改动。** 提交要勤、要聚焦。一次提交一件事，回退和排查才方便。

**直接在主分支上让 AI 做高风险改动。** 开个分支试，主分支留作兜底。

**提交信息随便写。** 让 AI 帮你写规范的，几乎零成本，长期回看时受益。

## **7. 小结**

Git 是 Vibe Coding 的安全绳。AI 改得越快越多，你越需要它来兜底——它是后悔药、记录仪、隔离带，让你敢放手让 AI 大改。会几个核心操作（status/add/commit/log）就够日常用，记不住的命令随时问 AI；用分支把 AI 的试验和稳定的主代码隔开；让 AI 帮你写规范的提交信息；最关键的，是 AI 改坏时能干净地退回去——细粒度用工具的 checkpoint、整段版本用 Git 的 reset/revert。

把 Git 这层兜底建立起来，你和 AI 的协作就有了底气：大胆让它试、坏了就退、退到干净状态重来。这份「随时能回去」的安全感，正是你敢真正放手用 AI 的底层支撑。
