让 Codex 自主干活，你心里多半有两个顾虑：它会不会乱改、乱跑命令，把东西弄坏？以及，难题它想得够不够深、简单活会不会用力过猛烧钱？前一个顾虑靠沙箱和审批控制，后一个靠推理强度和模型选择调节。这一篇就把这两组开关讲透。

把它们用顺，你就能在「安全可控」和「放手高效」之间，按每个任务的性质找到合适的挡位——既不会因为太谨慎而被频繁打断，也不会因为太放手而担惊受怕。

## **1. 两层安全模型**

Codex 的安全控制是两层独立的开关，理解这一点是后面一切的基础：**沙箱模式控制它技术上能做什么，审批策略控制它做之前要不要问你。**

沙箱管的是能力边界——它能在哪些地方写文件、能不能联网。审批管的是干预时机——在它越界或做有风险的事之前，要不要停下来征求你同意。这两层是正交的：你可以让它在一个宽松的沙箱里干活、但每步危险操作都先问你；也可以把它关在严格的只读沙箱里、却不打断它（因为只读本就没什么可批准的）。先建立这个「能做什么 × 何时问你」的二维认知，再看具体选项就清楚了。

## **2. 三种沙箱模式**

沙箱模式决定 Codex 技术上能在多大范围内动手，共三种，放手程度递增。

| 沙箱模式 | 它能做什么 |
|---|---|
| `read-only` | 只能读文件，不能改、不能写。最安全 |
| `workspace-write` | 读文件、在工作区内改文件、跑常规本地命令。日常默认 |
| `danger-full-access` | 去掉文件系统和网络边界，完全放开。风险最高 |

`read-only` 适合让它纯分析、审查、解释代码——它碰不了你的文件，你可以完全放心。`workspace-write` 是日常干活的低摩擦默认值——它能在你的工作区里改代码、跑测试这类常规命令，但出了工作区边界、或要联网就受限。`danger-full-access` 拆掉了所有边界（包括网络），只在你确实需要它完全放开时才用，平时别碰——一旦放开，恶意依赖联网外泄这类风险也跟着来了。

## **3. 三种审批策略**

审批策略决定 Codex 在动手前什么时候停下来问你，也是三种。

| 审批策略 | 什么时候问你 |
|---|---|
| `untrusted` | 只有已知安全的读操作自动跑，会改状态或触发外部执行的都要你批准 |
| `on-request` | 默认在沙箱内自主干，需要越过沙箱边界时才问你 |
| `never` | 从不弹审批，配合沙箱自主连续干 |

`untrusted` 最谨慎，几乎任何有副作用的动作都要你点头，适合你对它还不放心、想紧盯着的时候。`on-request` 是均衡之选——它在沙箱允许的范围内自己干，只有要做超出沙箱边界的事才停下来问你，既不频繁打断、又守住了关键关口。`never` 完全不打断，适合非交互的自动化场景（脚本、批处理），但它必须配合合适的沙箱用——既然不问你，就得靠沙箱把它能干的事限定好。

## **4. 怎么组合**

两层正交，组合出不同的工作姿态。几个常用搭配：

日常本地开发，用 `workspace-write` 沙箱 + `on-request` 审批——它能在工作区自主改代码跑测试，只在越界时问你，顺手又安全，这是大多数时候的默认。纯分析、审查代码，用 `read-only` 沙箱——它碰不了文件，审批怎么设都无所谓，省心。非交互的自动化（比如脚本里批量跑），用 `never` 审批，但沙箱要相应收紧（比如限定在 `workspace-write`），用沙箱代替人来兜底。至于 `danger-full-access` + `never` 这种全放开又不问的组合，威力最大但风险也最大，只在你完全清楚后果、且环境隔离良好时才用。

记住一个原则：**放手程度越高，越要靠沙箱兜底。** 你越是不想被打断（审批放宽），就越该把沙箱收紧，让它即便自主干也出不了边界。前面 AGENTS.md 那篇讲过的 Profiles 正好用在这里——把「日常」「只读分析」「放手自动化」几套沙箱+审批组合存成 profile，`--profile` 一键切换。

## **5. 推理强度 reasoning effort**

讲完安全，再看算力这组开关。Codex 支持调节推理强度（reasoning effort），从低到高大致是 `minimal`、`low`、`medium`、`high`、`xhigh`（最高档是否可用取决于模型）。强度越高，它在难题上想得越深、越周全，但也越慢越贵。

用法很直接：简单、套路化的活（补全、改个小函数、格式调整）用低档，又快又省；复杂、需要深度思考的活（架构设计、棘手 bug、性能优化、绕来绕去的逻辑）调到高档，让它多想一会儿把问题真正啃下来。别用最高强度干最简单的活（白烧钱），也别用最低强度硬刚难题（想不透）。这和前面 Cursor 篇讲的「按任务难度换模型」是一个内核——按活的难易匹配算力。

## **6. 模型选择与成本权衡**

Codex 当前这一代跑在 GPT-5.5 上，专为智能体任务训练。模型本身的选择不像 Cursor 那样有一长串多家可选，但「按活的价值匹配投入」的思路一样适用——推理强度就是你最主要的成本旋钮。

实战里把这几个开关一起考虑：日常大量的常规编码，用默认模型 + 中低推理强度，跑得快、花得少；真正难、价值高的任务，才把推理强度拉到 high 或 xhigh，让它充分思考。一个常见的省钱策略是「先低后高」：一个任务先用较低强度跑一遍，结果满意就省了；卡住了再调高重试。这样大部分活用低成本解决，只有少数硬骨头才动用高算力。把推理强度和前面的沙箱、审批一起，按任务的难度、风险、价值综合设挡，就是用好 Codex 的精髓。

## **7. 常见问题**

**Q：沙箱和审批到底什么关系？**
两层独立：沙箱管它技术上能做什么（在哪写、能否联网），审批管它做之前要不要问你。可自由组合。

**Q：日常该用什么配置？**
`workspace-write` 沙箱 + `on-request` 审批是低摩擦又安全的默认。纯看代码用 `read-only`。

**Q：把审批设成 never 安全吗？**
取决于沙箱。`never` 不问你，所以必须靠沙箱把它能干的事限定好。别把 `never` 和 `danger-full-access` 随便组合用。

**Q：推理强度越高越好吗？**
不是。高强度更慢更贵，只该用在难题上。简单活用低档省钱省时，按任务难易匹配才对。

## **8. 小结**

用好 Codex，本质是同时拧好两组旋钮。安全这组是两层：沙箱模式（`read-only` / `workspace-write` / `danger-full-access`）定它技术上能做什么，审批策略（`untrusted` / `on-request` / `never`）定它何时问你，两层正交、按「越放手越收紧沙箱」的原则组合，日常用 `workspace-write` + `on-request` 即可。算力这组是推理强度：从 `minimal` 到 `xhigh`，按任务难易匹配，简单省、难题深想。

把这两组旋钮按每个任务的风险和难度设到合适挡位，再用 Profiles 把常用组合一键切换，你就能让 Codex 既安全可控又放手高效——这正是驾驭一个自主代理的核心功夫。
