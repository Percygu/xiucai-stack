用 Cursor 的人里，效率分水岭往往不在于会不会用 Agent，而在于知不知道什么时候不该用 Agent。同样一个对话框，有人不管什么活都直接丢给 Agent 让它自己搞，结果简单的问答被它改了一堆文件，复杂的任务又因为没先对方向而跑偏；而会用的人，会根据这次要干的活，在几种模式之间切换——只想问问题时用只读的模式，复杂任务先让它出计划，确认了再放手执行。

Cursor 把这套「按需放手」的能力做成了四种工作模式：Ask、Agent、Plan、Edit。这一篇把它们逐个讲透：各自管什么、边界在哪、怎么切换，以及怎么把它们串成一套既快又稳的工作流。

## **1. 为什么要分模式**

让 AI 帮你写代码，不同的活需要不同的放手程度。你只是想问一句「这段逻辑是干嘛的」，根本不希望它改任何代码；你要做一个跨好几个文件的新功能，又希望它能自己规划、自己动手；你只想把眼前这个函数改个写法，又不需要它满项目乱翻。

模式就是用来匹配这种差异的。它本质上是在调两个旋钮：**能不能改代码**，以及**改的范围有多大**。Ask 完全不改，Edit 改你指定的一小块，Agent 放手改一整片，Plan 则在动手前先把方案定下来。理解了这两个旋钮，四种模式的定位就一目了然。在 Agent 输入框里按 `Shift+Tab`，就能在这几种模式之间切换。

## **2. Ask 模式**

Ask 是只读的对话模式：它能读你的代码、回答问题、给建议、讨论方案，但不会改动任何文件。

它适合那些你只想「问」、不想「动」的场景：搞懂一段陌生代码是干嘛的、让它解释某个报错的原因、和它讨论一个功能该怎么设计、请它给几个实现方案对比。因为它碰不了代码，你可以放心大胆地问，不必担心它顺手改了什么。

很多新手忽略 Ask，遇到什么都用 Agent，结果一个简单的「这个函数为什么这么写」也触发它去翻一堆文件甚至动手改。养成习惯：只要你这一轮的目的是理解、是问、是讨论方案，就用 Ask。把它当成一个读过你全部代码、随叫随到的明白人。

在 Ask 里几类提问特别值：让它讲清一段代码（「`@auth.js` 这个文件的登录流程是怎么走的」）、对比方案（「这个状态管理用 Context 还是 Redux，各自利弊」）、定位思路（「这个接口偶尔超时，可能的原因有哪些」）。注意它只给分析和建议，真要动手改时再切到 Edit 或 Agent。

## **3. Agent 模式**

Agent 是 Cursor 的默认模式，也是最强的——你给它一个目标，它自己读懂代码库、规划步骤、跨多个文件改动、在终端跑命令、看结果再迭代，整个过程基本自主完成。

它适合的是完整的、要动手的任务：实现一个新功能、修一个涉及多处的 bug、做一次跨文件的重构。你描述清楚要什么，它就一条龙地去做，你在旁边看它读哪些文件、改哪些地方、跑什么命令，最后验收。它的每一步改动都以 diff 呈现，你可以逐处接受或拒绝，不满意整体回退，所以放手让它干并不等于失控。

Agent 强归强，但有个前提：你得把要什么说清楚。目标含糊，它自主性越高就越容易朝你没料到的方向使劲。这正是 Plan 模式存在的理由——复杂任务先让它把方案摆出来对一遍，再让 Agent 放手干。

## **4. Plan 模式**

Plan 模式是近来 Cursor 很看重的一环——官方自己做新功能，现在大多也先让 Agent 写一份计划再动手，因为这明显提升了最终生成代码的质量。它解决的就是 Agent 在复杂任务上容易跑偏的问题。

进入 Plan 模式（在 Agent 输入框按 `Shift+Tab`，描述复杂任务时 Cursor 也会主动建议切到它），它会做三件事：先研究你的代码库，找到相关文件、查阅文档、必要时反问你澄清问题；然后生成一份 Markdown 格式的计划，里面带具体的文件路径和代码引用、列出要做的待办；接着你可以直接编辑这份计划，增删待办、调整方向。计划敲定后，再让它照着执行。

这套机制的价值在于把纠偏提前到了动手之前。你审的是一份结构化的计划，而不是一堆已经改完的代码——发现方向不对，改两行计划就行，成本极低。复杂的、跨多文件的、有架构取舍的任务，都该先走一遍 Plan。这和 Claude Code 的 Plan 模式是同一个思路。

## **5. Edit 模式**

Edit 是受控的局部编辑模式，最常见的入口就是前一篇讲的 `Cmd+K`：选中一段代码、按下它、用自然语言说要怎么改，它就只改这一段，不会满项目乱动。

它适合那些范围明确、你心里有数的小修改：把这个函数改成异步、给这段加错误处理、把这个循环改成更高效的写法。和 Agent 的区别在于「范围」——Agent 会自己判断要动哪些文件，Edit 只动你选中的、你指定的那一块。当你很清楚要改哪里、只想要一次精准的局部修改时，Edit 比 Agent 更快也更可控。

用 Edit 的诀窍是「选准 + 说清」：先把要改的那段代码选中（范围别多也别少），再用一句明确的指令说要改成什么样，比如「把这个回调改写成 async/await」「给这几行加上 try/catch 并打日志」。改完它就地给出 diff，你确认接受。范围越聚焦、指令越具体，结果越准——这也是为什么局部精修用 Edit 往往比让 Agent 去全局找更省心。

## **6. 推荐工作流**

四种模式不是各用各的，串起来才威力最大。一套稳妥又高效的节奏是：**先 Ask 或 Plan 想清楚，再 Agent 放手做，最后用 Edit 精修。**

具体来说：面对一个新任务，先用 Ask 把不清楚的地方问明白、或直接用 Plan 让它研究代码库出一份计划；你审一遍计划、纠正方向；确认后切到 Agent，让它照着计划放手实现，跨文件改动、跑测试、迭代；它做完后，哪里还差点意思、想微调某个细节，用 `Cmd+K`（Edit）精准改那一小块。这样既享受了 Agent 的自主高效，又用 Plan 把住了方向、用 Edit 兜住了细节。

简单的活当然不用这么全套——问一句话用 Ask、改一小块用 Edit、清晰的小功能直接 Agent。但凡是复杂、跨文件、有取舍的任务，「先 Plan 后 Agent 再精修」这条线几乎总是对的。

## **7. 四种模式怎么选**

把选择标准浓缩成一句话：**这一轮你想让 AI 改代码吗？想改多大范围？**

不想改、只想问或讨论，用 Ask。想改、但只改眼前指定的一小块，用 Edit。想改、范围较大要跨文件、且任务清晰，直接用 Agent。想改、但任务复杂、需要先对齐方向，先用 Plan 出方案再转 Agent。养成下意识先问自己这两句的习惯，你就不会再犯「什么都丢给 Agent」的错。

## **8. 常见问题**

**Q：四种模式怎么切换？**
在 Agent 输入框按 `Shift+Tab` 循环切换。Edit 也可以直接用 `Cmd+K` 在选中代码上唤起。

**Q：Agent 和 Plan 到底差在哪？**
Plan 只出计划不动手，让你先审方向；Agent 真正执行。复杂任务的标准做法是先 Plan 再 Agent，而不是二选一。

**Q：什么时候该退回用 Ask？**
当你这一轮的目的其实是理解、是问、是讨论方案，而不是要改代码时。用 Ask 能避免它多此一举地动你的文件。

**Q：Agent 改多了不放心怎么办？**
它的改动都是 diff，可逐处接受或拒绝、整体回退。复杂任务先 Plan 对方向，能从源头减少跑偏。

## **9. 小结**

Cursor 的四种模式，本质是给「让 AI 改代码」这件事配了不同的挡位：Ask 只读、Edit 局部、Agent 全自主、Plan 先规划。选对挡位的标准只有两条——要不要改、改多大范围。

真正高效的用法不是永远用最强的 Agent，而是按活的性质换挡：问问题退回 Ask，改一小块切 Edit，复杂任务先 Plan 再 Agent、最后用 Edit 收尾。把这套「按需放手」的节奏练成本能，你就从只会丢任务给 AI，进阶到了真正在驾驭它。
