到这里，你用 Cursor 干活的方式都还是「人在电脑前盯着」：你下指令、看它改、验收。但有些场景下，你其实不需要全程守着——交代清楚一个任务，让它自己在云端慢慢做，做完通知你就行；或者每次开 PR 时，有个东西能自动帮你审一遍代码、把问题指出来。

这正是 Cursor 云端代理和 BugBot 要解决的。它们把 AI 从「你身边的助手」扩展成「云端替你干活、自动把关质量的同事」。这一篇讲透这两块：云端代理怎么用、和本地 Agent 怎么选，以及 BugBot 怎么自动审 PR、甚至自动修。

## **1. 云端代理是什么**

前面讲的 Agent，都跑在你本地这台电脑上、占着你的编辑器。云端代理（Background Agents / Cloud Agents）则不同——它跑在 Cursor 云端的虚拟机里，独立、异步地干活，不占你的本地资源，也不需要你的编辑器一直开着。

它的典型用法是：你给它一个任务（实现某个功能、修某个 bug），它在云端的一台独立虚拟机里自己干，读代码、改文件、跑测试，干完直接开一个分支、提一个 PR 给你审。整个过程你不用守着，可以去干别的，它在背后推进，完成了通知你。这等于你多了一个能并行干活、还不抢你机器的远程同事。

## **2. 从哪里启动云端代理**

云端代理一个很方便的地方是入口多，不一定非得在编辑器里。你可以从 Cursor 的网页端（在浏览器里给它派任务、看进度）、从 Slack（在频道里 @Cursor 直接交代一个任务）、从手机 App 上启动它。

这意味着灵感或需求来的时候，你随手就能派活：开会时想到一个要改的地方，在 Slack 里 @Cursor 说一句让它去做；通勤路上在手机上瞄一眼某个 PR、派个代理去修。它在云端跑，你在哪、用什么设备都能指挥。

要让云端代理跑起来，前提是把它和你的代码仓库连上——它需要拉取你的仓库、在一个独立分支上干活、最后把分支推回去开 PR，所以第一次用要授权它访问对应的 GitHub 仓库。连好之后，派任务时把要做什么说清楚（和你给本地 Agent 下指令一样，越具体越好），它就在云端 VM 里基于最新代码开干。任务的边界越清晰，云端代理跑出来的结果越靠谱——因为你不在旁边随时纠偏，一开始就说明白比什么都重要。

## **3. 本地 Agent 还是云端代理**

两者都能干活，怎么选？看这个任务你要不要盯着、急不急。

需要你边看边调、即时反馈的活——探索式开发、一边讨论一边改、要频繁验收的——用本地 Agent，它就在你眼前，改一步看一步最顺手。可以交代清楚、放手让它慢慢做、不用你守着的活——一个边界明确的功能、一批机械的重构、一个你已经想清楚怎么修的 bug——丢给云端代理，你去干别的，它在云端跑完提 PR 你再审。

一个朴素判断：**要互动的留本地，能放手的丢云端。** 而且云端代理能并行——你可以同时派好几个代理各做一个独立任务，自己腾出手来。这是单纯本地干活给不了的吞吐量。

## **4. BugBot 自动审查 PR**

BugBot 是 Cursor 另一个很能打的能力：自动审查你的 Pull Request。你开一个 PR，它就自动跑起来——分析这次改动的 diff、理解你这次想干什么，然后在出问题的确切位置留下行内评论，把潜在 bug、风险点指出来，还给修改建议。每次 PR 有更新它会重新审，你也能手动触发。

它的实用性来自两个数字：审查速度快（一次审查平均一分多钟完成），命中率高（被它标出来的问题，超过七成会在 PR 合并前被开发者修掉）。这说明它指出的不是没用的噪音，而是真值得改的问题。对团队来说，它等于一个不知疲倦、每个 PR 都过一遍的自动审查员，把很多本该人工 review 才能发现的问题提前挡在合并之前。

启用 BugBot 要把 Cursor 的 GitHub 应用装到你的仓库上，授权它读取 PR、留评论。装好后，它就在每个新 PR 上自动跑，无需你每次手动触发；当然你也能在某个 PR 下手动叫它再审一遍。值得一提的是它审查的方式：它不是按固定套路逐项扫，而是像一个真正的审查员那样，针对这次 diff 动态地推理——该深挖的地方深挖、该调工具查上下文就去查，所以它能理解你这次改动的意图，而不只是机械地挑语法毛病。这也是为什么它的评论往往切中要害，而不是一堆模板化的提醒。

## **5. BugBot Autofix 与本地预审**

BugBot 不止能指出问题，还能动手修。它的 Autofix 能力会针对发现的问题，自动派出云端代理在独立虚拟机里测试你的改动、生成修复——这正是前面云端代理和 BugBot 的结合：审出问题，再用云端代理把它修了，形成「发现 → 修复」的闭环。这些自动修复有相当一部分会被直接合进原 PR，等于把一部分修 bug 的活也自动化了。

如果你想在代码推上去之前就先过一遍，Cursor 还提供了 `/review` 命令——在本地推送前运行它，会跑一遍 BugBot 和安全审查，相当于一道推送前的质量门禁。把它当成提交前的最后一道自检：本地先审一遍、改掉问题再推，比推上去等 PR 审查再回头改更高效。

## **6. 常见问题**

**Q：云端代理和本地 Agent 是一回事吗？**
不是。本地 Agent 跑在你机器上、要你盯着；云端代理跑在 Cursor 云端 VM、异步独立干活、干完提 PR。能放手的任务交云端，要互动的留本地。

**Q：云端代理需要我一直开着电脑吗？**
不需要。它在云端跑，你关了编辑器、合上电脑都不影响，完成后会通知你、并以 PR 形式交付。

**Q：BugBot 会不会乱报一堆没用的问题？**
它的命中率不低——被标出的问题超七成会被开发者在合并前修掉，说明大多是真问题。当然你仍是最终判断者，对它的评论照常审阅取舍。

**Q：BugBot 收费吗？**
它属于 Cursor 的付费能力，具体计费和额度随官方政策变动，用之前在 Cursor 账户里确认当前规则。

## **7. 小结**

云端代理和 BugBot，把 Cursor 从「你身边的编辑助手」推向了「云端替你干活、自动把关质量的协作者」。云端代理让你能交代清楚一个任务、放手让它在云端异步跑完提 PR，还能从 Web、Slack、手机随手启动、并行派多个；BugBot 让每个 PR 都自动过一遍审查、在出问题处留行内评论，配合 Autofix 还能自动派云端代理把问题修了。

用好它们的关键，是分清哪些活该「盯着做」、哪些活能「放手交出去」。要互动的留本地 Agent，能放手的丢云端代理；质量把关交给 BugBot 自动跑，自己只做最终裁决。这样你就从一行行盯着 AI 改，升级到了调度一支既在本地、又在云端的 AI 队伍。
