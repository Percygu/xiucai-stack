AI 答得准不准，八成取决于你喂给它的上下文。同一个问题，你只甩一句「这个为什么报错」，它只能凭猜；你把相关的文件、报错日志、用到的库文档一并指给它，它就能精准定位。很多人用 Cursor 觉得它「不够聪明」，其实是没把该给的上下文给到。

Cursor 在这件事上提供了两套互补的能力：一套是你主动用 `@` 引用、把具体的代码和资料喂进对话；另一套是 Memories，让它自动记住你这个项目的约定和偏好、跨会话都不忘。这一篇把这两套讲透，再加一组上下文管理的实用技巧。

## **1. 上下文决定 AI 的上限**

先建立一个核心认知：AI 不是无所不知，它只知道你给它的上下文里有的东西。你的项目代码、你用的库的最新文档、你团队的约定，这些它默认都不掌握。喂得越准、越全，它答得越对；喂得不够，它就只能基于训练里的通用知识猜，猜错了你还以为是它笨。

所以用 Cursor 的高手，花很多心思在「给对上下文」上。这不是麻烦事，而是最高杠杆的动作——一次精准的引用，胜过反复纠正它十次。

## **2. @ 引用体系**

`@` 是 Cursor 里最该用熟的符号。在对话框里打一个 `@`，会弹出一个菜单，让你把各种上下文精准地引用进来。几个最常用的类型：

`@文件` 引用某个具体文件的内容；`@文件夹` 引用整个目录；`@Code` 引用某个函数、类等具体符号；`@Git` 引用某次提交或某段改动，让它基于「这次改了什么」来回答；`@Web` 让它联网搜索最新信息；`@Docs` 引用某个库的官方文档（下面专门讲）。此外，你还能让它检索整个 codebase——当你不确定相关代码在哪时，它会自己在代码库里搜出相关片段。

这套 `@` 引用和 Claude Code 的 `@` 是同一个思路：与其用文字描述「我那个处理登录的文件」，不如直接 `@auth.js` 把它指给 AI。养成习惯——提问前先想一下「这个问题需要它看到哪些东西」，把那些用 `@` 引进来，回答质量会立刻上一个台阶。

举个对比就清楚。不带引用的问法，AI 只能猜：「帮我改下登录逻辑，加个失败重试」。带上精准引用的问法，它直接对着真实代码动手：「`@auth/login.ts` 给这里的登录请求加 3 次失败重试，参考 `@utils/retry.ts` 里已有的重试封装，错误格式按 `@Docs` 里的 API 文档来」。后者把「改哪个文件、参考哪段已有代码、遵循哪份文档」全指明了，它几乎不会跑偏。一次这样的精准引用，省下的是你来回纠正它好几轮的功夫。多个 `@` 可以在一句话里叠加使用，把这次任务相关的代码、参考、文档一并喂全。

## **3. @Docs 让 AI 用上最新文档**

`@Docs` 值得单独讲，因为它治的是 AI 一个高频毛病：用过时的 API。AI 的知识有截止时间，你用的库可能已经更新了、某个方法的用法变了，它却还按老的写，结果生成的代码跑不通。

`@Docs` 的解法是：你把某个库的官方文档 URL（比如 React、Next.js、Tailwind 的文档）添加进来，Cursor 会下载并索引它，之后你在对话里用 `@Docs` 引用，AI 回答和生成代码时就以这份最新文档为准，而不是凭训练里的旧记忆。遇到那种「AI 老是写出过时写法」的库，把它的官方文档加进 `@Docs`，这个问题基本就解决了。

## **4. Memories 自动持久记忆**

`@` 引用是你主动喂上下文，Memories 则是让 Cursor 自动记住、跨会话都不忘。它是一个由 AI 自己维护的持久知识库——在你和它协作的过程中，它会把你这个项目的约定和偏好慢慢学下来、记进去，下次新开对话，这些它还记得，不必你再重新交代。

这正是它取代旧版 Notepads（已在 2025 年底废弃）的原因：Notepads 要你手动建、手动写，Memories 则是自动积累。比如你在一次对话里纠正它「这个项目状态管理统一用 Zustand」，它就可能把这条记进 Memories，之后再写状态相关代码时自动遵守，你不用每次都强调。

## **5. 怎么用 Memories**

Memories 多数时候自动工作，但你也能主动管理它，两种方式。

一是在对话里直接告诉它记住：说一句「记住，这个项目所有日期都用 UTC 存储」，它就会把这条存进记忆，之后自动遵守。这是最自然的方式——干活过程中随时把重要约定一句话沉淀下去。二是在设置里手动管理：打开 Settings 里的 Memories，可以查看它记了哪些、手动添加或删除条目。如果发现它记错了、或某条过时了，进去删掉即可。

## **6. 与 Rules 和 CLAUDE.md 的区别**

这三个都和「让 AI 记住项目信息」有关，容易混。一句话分清：**Rules 是你主动写死的规矩，Memories 是 AI 自动攒下的记忆。**

Rules（上一篇讲的 `.cursor/rules`）是你明确写下的、要求 AI 必须遵守的硬约定，结构化、可提交 git、团队共享——适合放确定的编码规范。Memories 是 AI 在协作中自己学到、自动维护的软记忆，更像它「越用越懂你」的积累，偏个人、偏经验。两者配合：把团队的硬规范写进 Rules，让 AI 在用的过程中把零散的偏好自动沉淀进 Memories。

至于 CLAUDE.md，它是 Claude Code 那边的对应物，和 Rules 同源。如果你跨工具用，前面说过可以用通用的 `AGENTS.md` 写一份共享约定。理解了它们各自的定位，就不会再纠结同一条信息该写哪——确定的硬规矩进 Rules，让 AI 自己积累的交给 Memories。

## **7. 上下文管理技巧**

最后几条让上下文始终干净有效的实用习惯。

**新任务开新对话。** 干一件和之前无关的新活，别在老对话里接着聊——旧上下文会干扰它。开个新对话，上下文清爽，它判断更准。这和 Claude Code 的 `/clear` 是一个道理。

**精准引用，别一股脑全塞。** 不是上下文越多越好。把无关的文件、超长的内容一股脑塞进去，反而会稀释重点、让它抓不住关键。引用真正相关的那几个文件、那段代码就够了。

**用 @Git 让它看懂「改了什么」。** 排查刚引入的 bug、写改动说明时，用 `@Git` 把相关提交或 diff 引进来，它就基于实际改动来分析，比你口头描述准得多。

**让它先复述理解。** 复杂任务前，可以让它先用自己的话复述一遍你的需求和它打算怎么做。复述对了再让它动手，能在最便宜的阶段发现理解偏差。

## **8. 常见问题**

**Q：@ 引用和直接把代码粘进去有什么区别？**
`@` 引用让 Cursor 直接读取那个文件/资料的真实内容，比你手动粘贴更准、更省事，也不容易粘错或粘漏。

**Q：AI 老是用某个库的过时写法怎么办？**
用 `@Docs` 把那个库的官方文档 URL 加进来并索引，之后引用它，AI 就以最新文档为准。

**Q：Memories 记错了或记了不该记的怎么办？**
去 Settings 的 Memories 里手动删掉对应条目即可。也可以在对话里直接纠正它。

**Q：Notepads 还能用吗？**
Notepads 已废弃，被 Memories 取代。新项目直接用 Memories，不必再找 Notepads。

## **9. 小结**

喂对上下文，是用好 Cursor 最高杠杆的能力。一套靠你主动出手——用 `@` 把相关文件、符号、改动、外部文档精准引进对话，尤其 `@Docs` 能治 AI 用过时 API 的老毛病；另一套靠 AI 自动维护——Memories 把你项目的约定和偏好跨会话记下来，越用越懂你。

把这两套和上一篇的 Rules 搭起来看：Rules 是你立的硬规矩，Memories 是 AI 攒的软记忆，`@` 引用是当下这次对话的精准投喂。三者配合，再加上「新任务开新对话、精准引用别乱塞」的习惯，你就能让 Cursor 始终基于最相关、最新、最懂你的上下文来干活。
