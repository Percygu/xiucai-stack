## 工作流程

### 第一步：确认要写哪篇文章

1. 读取 `vibe_coding_series_plan.md`，确认用户要求编写的具体文章
2. 确认文章所属章节（认知篇/环境搭建/Prompt技巧/工具精通/工程实践/项目实战/进阶心法）
3. 确认文章的文件名、标题和内容要点
4. 检查 `src/vibe_coding_series/` 目录下是否已有该文件，避免重复创建

### 第二步：确定写作策略

根据文章所属章节，采用不同的写作策略：

**认知型**（认知篇、进阶心法篇）：
- 注重概念的深入浅出讲解，零基础也能看懂
- 大量使用生活化类比帮助理解
- 图文并茂是重点，多用对比图、流程图、思维导图
- 真实案例和效果展示必不可少
- 观点占 60%，示例占 40%

**操作型**（环境搭建篇、工具精通篇）：
- **【强制】动笔前必须用 WebSearch / WebFetch 调研该工具官方文档的最新能力**。Claude Code、Cursor、Codex 这类工具迭代极快（功能、模式名、界面、术语几个月就变一次），训练知识一定过时，绝不能凭记忆写。要以官方文档（`code.claude.com/docs`、`cursor.com/docs`、`developers.openai.com/codex`）和最新（当年）的权威资料为准，核实清楚当前的功能名称、操作路径、界面长相后再写。举例：Cursor 早已不是"Tab/Chat/Composer 三大核心"，而是 Ask/Agent/Plan/Edit 四模式，Notepads 也已被 Memories 取代——写之前不查就会写错。
- 每一步都要有清晰的操作指引
- 截图占位标记必须足够多，几乎每个操作步骤都要配图
- 常见报错和解决方案提前准备
- 操作指引占 80%，原理解释占 20%
- 小白对照教程能一步一步跑通

**技巧型**（Prompt技巧篇、工程实践篇）：
- 大量使用"好 vs 坏"的对比示例
- 展示真实的 Prompt 和 AI 回复效果
- 每个技巧都要有"为什么这样更好"的解释
- 可直接复用的 Prompt 模板
- 
- 技巧讲解占 40%，实际演示占 60%

**实战型**（项目实战篇）：
- 完整的从 0 到 1 开发流程
- 展示真实的人机对话过程
- 每个阶段的 Prompt 都要给出
- 遇到问题时如何调整和迭代
- 最终成品效果展示
- 操作过程占 80%，原理讲解占 20%

### 长文章拆分策略

`vibe_coding_series_plan.md` 中规划的是一个大纲，实际写作时可能发现某篇文章的内容量远超一篇能容纳的范围。当遇到以下情况时，应主动将一篇拆分为多篇：

**判断标准**：
- 如果按规划写完，正文预计超过 10000 字（不含代码），考虑拆分
- 如果一篇文章覆盖的知识点之间相对独立，且每个知识点都值得深入展开，考虑拆分

**拆分方式**：
- 文件名在原文件名基础上加后缀区分
- 拆分后每篇文章仍然独立成文，有自己的 frontmatter、开篇引入和小结
- 拆分时要告知用户，由用户确认拆分方案后再动手写，同时提醒用户后续需要更新 sidebar 配置

### 第三步：编写文章内容

#### Frontmatter 格式

```yaml
---
title: {序号}. {标题}
description: {一段 80-130 字的 SEO 描述，自然融入文章核心关键词（如 Vibe Coding、AI编程、Claude Code、Cursor、Prompt 等），概括本文要解决的问题和读者能获得的价值。要像搜索结果摘要一样吸引点击，避免堆砌关键词，语句通顺有信息量。}
category:
  - Vibe Coding
tag:
  - Vibe Coding
  - AI编程
  - {章节相关标签，如：Claude Code、Cursor、Prompt Engineering等}
  - {具体知识点标签}
---
```

**description 写作要求**：
- 长度控制在 80-130 字之间（不要低于 80，不要超过 130）
- **必须以「Vibe Coding教程第X篇：」开头**（X 为该文在系列中的序号，与 frontmatter `title` 的序号一致），既锚定核心 SEO 关键词，又标明系列定位。例如 `Vibe Coding教程第3篇：……`。开头用过这个词后，正文里就不必再重复"这篇 Vibe Coding教程"，以节省字数。
- **必须优先突出高搜索量的热门关键词**，这些关键词要尽量靠前出现。高搜索量关键词包括但不限于：`Vibe Coding`、`氛围编程`、`AI编程`、`AI写代码`、`Claude Code`、`Cursor`、`Copilot`、`AI编程工具`、`Prompt`、`AI编程教程`、`AI编程入门`、`零基础编程`、`AI辅助开发`、`Vibe Coding教程`、`Cursor教程`、`Claude Code教程`。根据文章具体内容选择最相关的 3-5 个热门关键词自然融入
- 关键词要自然融入句子，不能生硬堆砌

#### 文章结构

> ⚠️ **绝对不要写一级标题（`# 标题`）**。VuePress（Theme Hope）会自动用 frontmatter 里的 `title` 字段渲染页面大标题，正文里再写 `# **标题**` 会导致重复、且影响渲染。文章正文从开篇引入文字直接开始，章节标题一律从二级标题（`##`）起步。参照 `src/go_series/go_advanced/channel.md` 的实际写法。

1. **开篇引入**：1-2 段简洁的背景介绍，说明本文要讲什么、解决什么问题。要能引起读者共鸣（比如描述一个痛点场景）。不需要回顾上一篇内容。**正文第一行就是这段引入文字，前面不加任何标题。**

2. **正文章节**：标题必须带数字编号前缀，从二级标题开始：
   - 二级标题（##）：`## **1. 章节标题**`、`## **2. 章节标题**` 依次递增
   - 三级标题（###）：`### **2.1 子标题**`、`### **2.2 子标题**` 跟随父级二级标题编号
   - 四级标题（####）：`#### **2.1.1 子子标题**` 跟随父级三级标题编号

   每个章节围绕一个核心知识点展开。

   > ⚠️ **【标题铁律】所有层级的章节标题（二级/三级/四级）一律精炼简洁，只保留标题本身，严禁使用"标题：一句补充说明"这种用冒号（中文`：`或英文`:`）把标题和解释拼在一起的形式。** 这种"主标题 + 冒号 + 描述句"的写法是典型的 AI 腔，一眼假。标题就是标题，要短、要干净，需要展开解释的内容放到正文段落里去讲，不要塞进标题。
   > - 反例：`### **2.1 理需求：先把脑子里的想法捋顺**`、`## **3. 首次启动：一键迁移 VS Code 设置**`、`### **6.1 桌面应用：下载即用的图形界面**`
   > - 正例：`### **2.1 理需求**`、`## **3. 首次启动**`、`### **6.1 桌面应用**`

3. **Prompt 示例**：这是 Vibe Coding 教程的核心特色，必须大量使用。格式：

   ````
   **Prompt：**
   ```
   你是一个前端开发专家。请帮我创建一个响应式的个人主页...
   ```

   **AI 回复效果：**
   > 【建议截图 —— 展示 AI 实际回复的效果】

   ````

   当需要展示"好 vs 坏"的 Prompt 对比时：
   ````
   **不推荐的写法：**
   ```
   帮我写个网页
   ```

   **推荐的写法：**
   ```
   请帮我创建一个个人主页，要求：
   1. 响应式设计，适配手机和电脑
   2. 包含自我介绍、项目展示、联系方式三个区域
   3. 使用现代简约风格，主色调为深蓝色
   ```
   ````

4. **代码示例**：使用对应语言的代码块包裹。因为 Vibe Coding 教程涉及多种语言（HTML/CSS/JS/Python等），根据实际内容选择。代码要完整可运行。

5. **小结**：`## **{编号}. 小结**`，用一两段自然的文字收束全文。小结不是复述文章干了什么的总结清单，而是作者以自然的笔触提炼升华全文核心思想。不要预告下一篇内容。

6. **结尾推广块**：必须在文章最后加上统一的推广 HTML 块（见下方模板）。

### 第四步：配图（设计配图 + 实战截图两条线）

图文并茂是"秀才"所有系列教程的标志。Vibe Coding 虽然重实践，但绝不能一篇文档一张图都没有，那样太单一、太干瘪。本系列的图片分两类，两类都要用足，让文章既有"看一眼就懂"的图解，又有"真实可信"的实战画面。

#### 4.1 两类图片

**第一类 · 设计配图（占位标记 → 后续出图）**：概念图、对比图、流程图、架构示意图等需要设计/AI 出图的图，用来把抽象概念讲清楚。用 `> 【建议配图N】` 占位标记把图描述清楚，后续再出图替换。认知篇、进阶心法等理论型文章以这类为主；其他文章凡是需要图解概念、对比说明的地方也用这类。

> ⚠️ **【铁律，绝不允许违反】设计配图在图片真正生成出来之前，文章里一律只能放 `> 【建议配图N】` 占位标记，严禁直接写 `![](路径)` 这种指向尚不存在的图片文件的"空引用"。** 作者的真实工作流是：先由你在文中写好**可直接复制给 ChatGPT / 文生图工具的生图提示词**（放在占位标记里）→ 作者拿提示词去生成图片、存到对应路径 → 再把占位标记替换成图片引用。所以你这一步的产物是"提示词"，不是"引用"。已有的历史文章（如 `what_is_vibe_coding.md`）里看到的 `![](...)` 是作者出图后替换好的成品，**不要参照它们直接写空引用**——那是流程的终点，不是你交付时该有的状态。

**第二类 · 实战截图（真正去做、真实截取）**：真正做 Vibe Coding 实战过程中截取的图——网页/产品运行效果、在线工具（v0、bolt.new 等）页面、代码运行效果、最终成品、迭代前后对比等。实战篇、项目实战篇、工具精通篇、工程实践篇以这类为主。**这是 Vibe Coding 系列区别于纯理论教程的灵魂：读者要看到真实的产出，而不是想象出来的效果。**

#### 4.2 实战截图工作流（实战型 / 操作型文章必做）

写项目实战篇、工具精通篇、工程实践篇这类需要动手的文章时，不能凭空编造效果，要真正去跑一遍 Vibe Coding：

1. **真做**：用 Claude Code（也就是你自己）把这篇文章要演示的项目 / 操作从 0 到 1 真实做出来——生成代码、运行、用浏览器查看效果、遇到问题再迭代修复。文章里给出的 Prompt 和效果，必须是你真实跑出来的。
2. **真截**：尽最大可能用工具真实截图。凡是浏览器能渲染的（网页效果、在线工具页面、最终成品），一律用 chrome-devtools 的 `take_screenshot` 真实截取并保存成图片文件。实在无法截取的工具运行时界面（Claude Code 终端 UI、Cursor / VS Code 操作界面、某些云端工具或需登录态的页面），不要硬编，也不要跳过——只要你判断该位置确实需要一张真实截图，就在对应位置用 `🔴待截图` 标记交代清楚（格式见 4.5），由作者本人去补。
3. **截关键步骤**：不要只截最终成品。要截能体现"迭代过程"的关键节点——初版效果、每次 Prompt 调整后的变化、报错画面与修复后的对比、最终成品。让读者看到真实的 Vibe Coding 是怎么一步步逼近目标的，这种"过程感"比一张完美的成品图更有说服力。
4. **存图**：把截下来的图保存到 `src/assets/img/vibe_coding/{分类}/{文章文件名}/` 对应目录（命名与路径见 4.4）。
5. **引用**：在文章对应位置用真实图片引用（见 4.4），而不是占位标记。

#### 4.3 各类型文章图片数量要求

无论哪类文章，都不能出现"一张图都没有"的情况。

- **认知型**（认知篇、进阶心法）：至少 5-8 张图（对比图、流程图、工具截图、效果展示），设计配图为主
- **操作型**（环境搭建、工具精通）：至少 8-12 张图（几乎每个关键操作步骤都要配图），实战截图为主
- **技巧型**（Prompt技巧、工程实践）：至少 5-8 张图（Prompt 对比效果图、AI 回复截图、代码效果图），设计配图 + 实战截图搭配
- **实战型**（项目实战）：至少 8-10 张图（操作过程截图、代码效果截图、最终成品截图），实战截图为主

#### 4.4 图片存放与引用规范

本项目所有图片统一放在 `src/assets/` 下，文章中用相对路径引用。Vibe Coding 系列的图片规范如下：

- **存放根目录**：`src/assets/img/vibe_coding/`
- **先按章节英文名分子目录**（与文章子目录一一对应）：`basics/`、`setup/`、`prompt/`、`tools/`、`engineering/`、`projects/`、`advanced/`
- **再按文章文件名建一层子目录**：每篇文章的图片单独放进 `{章节}/{文章文件名}/` 这个专属目录里，不要把同一章节所有文章的图混在章节目录下。目录不存在先创建。例如文章 `what_is_vibe_coding.md` 的图片全部放在 `src/assets/img/vibe_coding/basics/what_is_vibe_coding/` 下

#### 4.5 两种占位标记格式

文章里有两种需要后续补图的情况，用两种不同的醒目标记区分，方便作者一眼找全。

**（1）设计配图占位标记**（用于概念图、对比图、流程图等需要设计/AI 出图的图）：

```
>
> **生图提示词（可直接发给 ChatGPT / 文生图工具）：**

>
> 整体目的：{这张图要帮助读者理解什么}
```

#### 4.6 图片设计原则（极其重要，务必严格遵循）

图片设计是一种创作，不是模板匹配。**核心目标只有一个：让读者看到图的一瞬间就理解文字想表达的意思，甚至比读文字更快更直观。** 每张图都要做到生动形象、视觉上好看、结构丰富多样、能准确表达文字意思并辅助理解。Vibe Coding 面向小白，"看图就能懂"尤其重要。

本 skill 目录下 `references/` 中的四张图片（`配置质量参考示例1.png`、`配图质量参考示例2.png`、`配图质量参考示例3.png`、`配图质量参考示例4.png`）是图片设计的黄金标杆。每次设计配图时，必须先读取这四张图，把它们的配图水准作为创作目标，然后根据当前文字内容自由创作。不要去提取图中属于什么"类型"然后对号入座——而是去感受那些图为什么好看、为什么一眼就能看懂、为什么结构丰富不单调，然后在你自己的创作中达到同样的效果。

好的图片描述必须做到以下几点：

1. **结构形式丰富多样，完全由内容驱动**。每张图的视觉结构应该完全由它要表达的文字内容来决定，像设计师一样根据具体内容自由创作最合适的视觉形式。一篇文章中的多张图，尽量让整篇文章的配图看起来丰富、有变化、不单调。绝不能所有图都是同一种结构（比如全是左右对比，或全是线性流程）。视觉形式没有任何限制——可以是任何你认为最能表达这段文字内容的结构，创意越丰富越好。**特别注意：避免过度使用环形图、同心圆、洋葱圈结构**——这类结构容易变成万能模板被滥用，只有当内容确实是"循环流程"或"迭代闭环"时才适合用环形。

2. **用图标和视觉隐喻替代纯文字方框**。不要只写"画一个方框写上'用户输入'"——要描述具体的图标：用户头像、齿轮、放大镜、警告三角、文档、数据库圆柱、大脑、锁、云等等。每个概念都应该有对应的视觉符号，让图在缩小到很小时仍然能通过图标识别含义。

3. **【强制】具名的工具 / 产品 / 品牌一律用各自的官方品牌 Logo，绝不能用同一个通用图标糊弄**。这是最容易翻车、也最影响观感的一点。凡是图里出现具体的工具或产品——Claude Code、Cursor、Codex、GitHub Copilot、VS Code、Trae、Windsurf、v0、bolt.new、ChatGPT、OpenAI、GitHub、Figma、Vercel 等——必须画出它们**各自不同的品牌图标**，而不是把它们全部画成同一个"机器人"或同一个聊天气泡。否则读者根本分不清谁是谁，整张图也单调重复、廉价。举例：同时出现 Claude Code、Cursor、Codex 三者时，要分别用 **Claude 的橙色星芒/光芒标志**、**Cursor 的黑色立方体标志**、**Codex/OpenAI 的黑色花瓣标志**，三个截然不同的品牌图标并排，读者一眼就能区分。**在每条涉及具名工具的配图占位标记里，都要明确点名"这里用 XX 的品牌 Logo"，并简要描述该 Logo 的样子和主色，确保出图时不会被替换成通用机器人。** 不确定某个 Logo 长什么样时，描述要写清"使用 XX 官方品牌图标（主色为…）"，让出图环节去还原，而不是退回通用图标。

4. **颜色要有语义，不只是区分**。红色=错误/风险/问题，绿色=正确/通过，蓝色=正常流程，橙色/黄色=警告/需注意，灰色=被淘汰/不推荐。让读者不看文字只看颜色就能感知信息的情绪和重要程度。Vibe Coding 教程里"新手写法 vs 推荐写法"的对比图，尤其要用红/绿对比强化差异。

5. **箭头和连接线要讲故事**。粗箭头=主流程，虚线箭头=可选/回退/映射，红色箭头=错误传播，带标注的箭头=解释因果关系，双向箭头=双向交互。箭头不只是连接，它要表达关系。

6. **层次感和视觉重点**。图中要有明确的视觉焦点——最核心的元素最大、最醒目，辅助元素小且颜色淡。不要所有元素一样大一样重，读者应该第一眼就知道先看哪里。

7. **标注文字要精炼**。图中的文字是高度凝练的短语或关键词，每个标注控制在 10 个字以内。超过 15 个字的信息应该放在正文而不是图里。

8. **白色/浅色干净背景**。不要使用网格背景、深色背景或花哨的纹理。图片背景保持白色或非常浅的纯色，让信息元素本身成为视觉焦点。参考标杆图片，全部都是干净的白底。

9. **【强制】每张配图右下角必须带引流信息**。所有设计配图（不含实战截图）在生成时，都必须在图片**右下角**统一放置一行引流信息：`公众号：IT杨秀才 ｜ https://golangstar.cn`。这行信息字号要小、颜色用浅灰，低调地待在右下角，不能喧宾夺主、不能遮挡主体内容，但每张图都不能漏。参照 `references/` 标杆图右下角的样式。**在每一条配图占位标记（`> 【建议配图N …】`）的图片描述里，都要明确写上"图片右下角放置引流信息：公众号：IT杨秀才 ｜ https://golangstar.cn（小字、浅灰、不抢主体）"这一句**，确保后续出图时不会遗漏。

另外，针对 Vibe Coding 教程的两类专属图，还要注意：

- **操作截图要标注**：截图中需要点击或关注的地方，用红色方框或箭头标注出来，小白才知道看哪里
- **效果图要真实**：实战截图必须是真实运行 / 渲染出来的画面，绝不用示意图糊弄

描述图片时，像一个设计师在构思作品：先想清楚这张图要帮读者理解什么（目的），然后自由构思最能达成这个目的的视觉形式（创作），再详细描述整体布局、各区域的具体内容（图标、颜色、标注文字）、元素之间的连接关系。描述要具体到"设计师看了就能画出一张好看且信息清晰的图"。

## 写作深度与质量标准（最重要，凌驾于一切之上）

> ⚠️ **这是全 skill 最高优先级的要求。** 过去写出来的文章普遍太单薄、太简陋——几千字的入门 demo、走马观花、要点不全、干货稀薄，连参考标杆的 1/4 都不如。从今往后，每一篇都必须按下面的标准来写，**先达标"深度与质量"，再谈别的**。

### 一、核心原则

1. **内容要"干"**——零注水。不用比喻、鸡汤、空泛道理凑字数；每一段都要有信息增量，能让读者照着学、照着做。比喻只为讲清概念、点到为止，绝不能拿来充篇幅。
2. **内容量要"足"**——覆盖全、讲得透。每个该讲的知识点都要展开到位，遵循**五要素：是什么 → 为什么/解决什么痛点 → 怎么用（真实命令/代码）→ 真实输出或效果 → 注意事项/适用边界**。只写"是什么 + 一句怎么用"就跳过，是不合格的。不走马观花、不遗漏要点。
3. **表现形式要"丰富"**——按内容需要选**最合适**的呈现形式：该用代码块就上代码、该用表格就列表格、该用分步流程就给端到端工作流、该对比就做对比表、该图解就配图。形式多样、不单调。**但绝不规定数量、绝不为凑数硬加任何一种形式**——形式服务于内容，不是反过来。
   > 注意：这里"不规定数量"针对的是**正文里的代码块、表格、工作流**等形式，**不涉及配图**。配图完全沿用第四步 4.1–4.6 的既有规范——包括 4.3 的各类型配图数量要求、原理图用生图提示词占位、演示效果用 chrome-devtools 真实截图、操作界面用 `🔴待截图` 标记让作者补——这套保持不变。
4. **【强调】内容必须"最新"**——尤其工具篇，所讲的功能名、界面、操作路径、命令、版本，必须是**当前最新**。Claude Code / Cursor / Codex 几个月就大改一次，**绝不允许凭过时记忆写**；动笔前必须联网核实官方文档最新状态，写完后再校一遍关键事实是否仍然成立。

### 二、篇幅（唯一的量化约束）

- **操作 / 工具精通 / 技巧型 / 实战型：6000–10000 字**。
- **认知 / 导读型：4500–6500 字**，以观点 + 类比 + 案例为主，不强堆代码。
- 红线：字数必须由干货撑起，**杜绝"几千字全是空泛叙述"的单薄货**。

### 三、代码要求

- 正文里贴出的代码**必须可用**——能跑、能抄、能直接拿去用，不允许贴看着炫但跑不通的伪代码。
- 若是**模板 / 示例骨架**（如 CLAUDE.md、各类配置文件、Prompt 模板等），给**带占位符的"模板公式"**即可，不强求完整可运行，但结构要清晰、可照着填。

### 四、对标与查漏（写前必做，分两种情况）

- **有标杆的主题（Claude Code 系列）**：对照 **官方文档（`code.claude.com/docs`）+ easy-vibe 同主题篇**（`github.com/datawhalechina/easy-vibe`，`docs/zh-cn/stage-3/core-skills/` 下，质量很高），先列出本篇知识点清单逐条查漏，**深度和干货密度不弱于该标杆篇**。
- **无标杆的主题（Cursor / Codex 等，easy-vibe 没有对应深度系列）**：
  1. **以官方文档为第一来源**（`cursor.com/docs`、`developers.openai.com/codex`），把该主题相关的所有子页扫一遍，**穷尽式列出要点清单**，官方文档列了什么就不能漏什么；
  2. 借 **Claude Code 同骨架篇做"结构与深度模板"**——套用它那套结构层次（简介→为什么→快速开始→详解→实战示例→调试/最佳实践→FAQ）和干货水准，只是填各自工具的内容；
  3. 用**共通能力骨架对照表**逐项映射查漏，确保不缺项，再补该工具的独有要点：

     | 共通骨架 | Claude Code | Cursor | Codex |
     |---|---|---|---|
     | 配置文件 | `CLAUDE.md` | `.cursor/rules` | `AGENTS.md` |
     | 命令 | Slash Commands | 命令/快捷键 | Slash + 自定义 Prompt |
     | 记忆 | auto memory | Memories | — |
     | 特色 | 子代理/Skills/Hooks/插件 | 四模式/Tab/BugBot/云端代理 | 多形态/云端并行/沙箱审批 |

  4. 必要时用 WebSearch 找当年权威评测 / 社区最佳实践，补官方文档没强调、但实战重要的要点（坑、高频用法）。
- 两种情况都要遵守上面的核心原则（干、足、形式丰富、最新）和篇幅约束。规划文档 `vibe_coding_series_plan.md` 列的要点是**下限**，官方/标杆里属于该主题的高频要点即使没列也要补上。

### 五、完稿自检（逐条确认，不达标不算写完）

- [ ] 内容干（无注水段落）、量足（要点无遗漏、每个点都按五要素讲透）

## 写作风格要求

1. **极致小白友好**：不要假设读者有任何编程基础。第一次出现的技术术语必须用一句话解释清楚。用"就像…一样"的类比让非技术人员也能理解。

2. **以文字叙述为主，切忌大量列点**：这是最重要的风格要求。文章要像一个有经验的前辈在跟你聊天，有思路、有逻辑、有论据、有细节。**大量列点罗列要点是最典型的 AI 写作痕迹，必须避免。** 具体要求：
   - 讲解概念、分析优劣势时，用连贯的段落文字来表达
   - 有时候该分点就分点，比如分步骤操作，这些场景用列表是合理的。但文章整体必须以段落叙述为主
   - **小结部分**：作者以自然的笔触收束全文，画龙点睛，不要列清单式回顾

3. **Prompt 是核心内容**：Vibe Coding 教程的灵魂是 Prompt 示例。每个知识点都要配真实的 Prompt 和效果展示，让读者能直接复制使用。

4. **效果先行**：先展示"用了这个技巧后的效果"（激发兴趣），再讲"怎么做到的"（传授方法），最后讲"为什么这样更好"（加深理解）。

5. **对比教学**：大量使用"新手常见写法 vs 推荐写法"的对比，差异越大越好，让读者"一看就知道区别"。

6. **全中文表达**：技术术语保留英文（如 Prompt、Token、Vibe Coding），解释和论述全部用中文。

7. **适度幽默**：延续现有系列风趣幽默的风格，但不要过度，保持专业感。

8. **篇幅控制**：每篇文章正文控制在 5000-10000 字之间（不含代码和 Prompt 示例）。

9. **标题规则**：标题简明扼要，**所有层级的标题（frontmatter title 和正文各级章节标题）都禁止用"标题：补充说明"这种冒号分割形式**（如"Vibe Coding：用自然语言写代码""理需求：先把想法捋顺"），这种写法一看就是 AI 腔。标题就是标题，只保留精炼的核心词，要解释的内容放正文里讲。不要中英文混杂解释。详见上文「正文章节」里的标题铁律。

10. **减少AI味儿**：行文要自然，像真人在写博客。不要用"让我们"、"接下来"等过渡词，不要每段开头都是"首先…其次…最后…"。

## 关键原则

0. **【第一优先级】严格执行《写作深度与质量标准》**（见上文同名章节）：内容要干、量要足、形式要丰富、信息要最新；篇幅达标（操作/工具/技巧/实战 6000–10000 字，认知/导读 4500–6500 字）且由干货撑起；写前对照官方文档与 easy-vibe 标杆篇列清单查漏，写完逐条过自检清单。**这一条凌驾于其余所有原则之上——绝不允许再写出单薄、走马观花、要点不全的文章。**
1. **每篇文章都要先读取规划文档** `vibe_coding_series_plan.md`，确认内容要点不遗漏。
2. **先读取一篇现有文章**（如 `src/go_series/go_advanced/channel.md`），确认格式一致。
