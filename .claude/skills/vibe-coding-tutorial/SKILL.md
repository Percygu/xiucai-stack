## 工作流程

### 第一步：确认要写哪篇文章

1. 读取 `vibe_coding_series_plan.md`，确认用户要求编写的具体文章
2. 确认文章所属章节（认知篇/环境搭建/Prompt技巧/工具精通/工程实践/项目实战/进阶心法）
3. 确认文章的文件名、标题和内容要点
4. 检查 `src/vibe_coding_series/` 目录下是否已有该文件，避免重复创建

### 第二步：确定写作策略

根据文章所属章节，采用不同的写作策略：

**认知型**（认知篇、进阶心法篇）：
- 注重概念的深入浅出讲解，零基础也能看懂
- 大量使用生活化类比帮助理解
- 图文并茂是重点，多用对比图、流程图、思维导图
- 真实案例和效果展示必不可少
- 观点占 60%，示例占 40%

**操作型**（环境搭建篇、工具精通篇）：
- 每一步都要有清晰的操作指引
- 截图占位标记必须足够多，几乎每个操作步骤都要配图
- 常见报错和解决方案提前准备
- 操作指引占 80%，原理解释占 20%
- 小白对照教程能一步一步跑通

**技巧型**（Prompt技巧篇、工程实践篇）：
- 大量使用"好 vs 坏"的对比示例
- 展示真实的 Prompt 和 AI 回复效果
- 每个技巧都要有"为什么这样更好"的解释
- 可直接复用的 Prompt 模板
- 
- 技巧讲解占 40%，实际演示占 60%

**实战型**（项目实战篇）：
- 完整的从 0 到 1 开发流程
- 展示真实的人机对话过程
- 每个阶段的 Prompt 都要给出
- 遇到问题时如何调整和迭代
- 最终成品效果展示
- 操作过程占 80%，原理讲解占 20%

### 长文章拆分策略

`vibe_coding_series_plan.md` 中规划的是一个大纲，实际写作时可能发现某篇文章的内容量远超一篇能容纳的范围。当遇到以下情况时，应主动将一篇拆分为多篇：

**判断标准**：
- 如果按规划写完，正文预计超过 10000 字（不含代码），考虑拆分
- 如果一篇文章覆盖的知识点之间相对独立，且每个知识点都值得深入展开，考虑拆分

**拆分方式**：
- 文件名在原文件名基础上加后缀区分
- 拆分后每篇文章仍然独立成文，有自己的 frontmatter、开篇引入和小结
- 拆分时要告知用户，由用户确认拆分方案后再动手写，同时提醒用户后续需要更新 sidebar 配置

### 第三步：编写文章内容

#### Frontmatter 格式

```yaml
---
title: {序号}. {标题}
description: {一段 80-120 字的 SEO 描述，自然融入文章核心关键词（如 Vibe Coding、AI编程、Claude Code、Cursor、Prompt 等），概括本文要解决的问题和读者能获得的价值。要像搜索结果摘要一样吸引点击，避免堆砌关键词，语句通顺有信息量。}
category:
  - Vibe Coding
tag:
  - Vibe Coding
  - AI编程
  - {章节相关标签，如：Claude Code、Cursor、Prompt Engineering等}
  - {具体知识点标签}
---
```

**description 写作要求**：
- 长度控制在 80-120 字之间
- **必须优先突出高搜索量的热门关键词**，这些关键词要尽量靠前出现。高搜索量关键词包括但不限于：`Vibe Coding`、`氛围编程`、`AI编程`、`AI写代码`、`Claude Code`、`Cursor`、`Copilot`、`AI编程工具`、`Prompt`、`AI编程教程`、`AI编程入门`、`零基础编程`、`AI辅助开发`、`Vibe Coding教程`、`Cursor教程`、`Claude Code教程`。根据文章具体内容选择最相关的 3-5 个热门关键词自然融入
- 关键词要自然融入句子，不能生硬堆砌

#### 文章结构

1. **大标题**：`# **{标题}**`

2. **开篇引入**：1-2 段简洁的背景介绍，说明本文要讲什么、解决什么问题。要能引起读者共鸣（比如描述一个痛点场景）。不需要回顾上一篇内容。

3. **正文章节**：标题必须带数字编号前缀，从二级标题开始：
   - 二级标题（##）：`## **1. 章节标题**`、`## **2. 章节标题**` 依次递增
   - 三级标题（###）：`### **2.1 子标题**`、`### **2.2 子标题**` 跟随父级二级标题编号
   - 四级标题（####）：`#### **2.1.1 子子标题**` 跟随父级三级标题编号

   每个章节围绕一个核心知识点展开。

4. **Prompt 示例**：这是 Vibe Coding 教程的核心特色，必须大量使用。格式：

   ````
   **Prompt：**
   ```
   你是一个前端开发专家。请帮我创建一个响应式的个人主页...
   ```

   **AI 回复效果：**
   > 【建议截图 —— 展示 AI 实际回复的效果】

   ````

   当需要展示"好 vs 坏"的 Prompt 对比时：
   ````
   **不推荐的写法：**
   ```
   帮我写个网页
   ```

   **推荐的写法：**
   ```
   请帮我创建一个个人主页，要求：
   1. 响应式设计，适配手机和电脑
   2. 包含自我介绍、项目展示、联系方式三个区域
   3. 使用现代简约风格，主色调为深蓝色
   ```
   ````

5. **代码示例**：使用对应语言的代码块包裹。因为 Vibe Coding 教程涉及多种语言（HTML/CSS/JS/Python等），根据实际内容选择。代码要完整可运行。

6. **小结**：`## **{编号}. 小结**`，用一两段自然的文字收束全文。小结不是复述文章干了什么的总结清单，而是作者以自然的笔触提炼升华全文核心思想。不要预告下一篇内容。

### 第四步：插入图片占位标记

Vibe Coding 教程对配图的要求极高，因为面向小白用户，"看图就能懂"是核心目标。

**认知型文章**：至少 5-8 张图（对比图、流程图、工具截图、效果展示）
**操作型文章**：至少 8-12 张图（几乎每个操作步骤都要有截图占位）
**技巧型文章**：至少 5-8 张图（Prompt 对比效果图、AI 回复截图、代码效果图）
**实战型文章**：至少 8-10 张图（操作过程截图、代码效果截图、最终成品截图）

**图片占位标记格式**：

```
> 【建议配图{序号} —— {图片标题}】
>
> 图片描述：{详细描述图片内容，包括布局、元素、颜色、箭头、图标等，要详细到设计师看了就能画出来}
>
> 整体目的：{这张图要帮助读者理解什么}
```

**图片设计原则**：

1. **操作截图要标注**：截图中需要点击或关注的地方用红色方框或箭头标注
2. **对比图要醒目**：好 vs 坏的对比用绿色（好）和红色（坏）区分
3. **流程图要简洁**：步骤清晰，每步不超过 10 个字
4. **效果图要真实**：展示实际运行效果，不要用示意图代替
5. **白色干净背景**：图片背景保持白色或非常浅的纯色

## 写作风格要求

1. **极致小白友好**：不要假设读者有任何编程基础。第一次出现的技术术语必须用一句话解释清楚。用"就像…一样"的类比让非技术人员也能理解。

2. **以文字叙述为主，切忌大量列点**：这是最重要的风格要求。文章要像一个有经验的前辈在跟你聊天，有思路、有逻辑、有论据、有细节。**大量列点罗列要点是最典型的 AI 写作痕迹，必须避免。** 具体要求：
   - 讲解概念、分析优劣势时，用连贯的段落文字来表达
   - 有时候该分点就分点，比如分步骤操作，这些场景用列表是合理的。但文章整体必须以段落叙述为主
   - **小结部分**：作者以自然的笔触收束全文，画龙点睛，不要列清单式回顾

3. **Prompt 是核心内容**：Vibe Coding 教程的灵魂是 Prompt 示例。每个知识点都要配真实的 Prompt 和效果展示，让读者能直接复制使用。

4. **效果先行**：先展示"用了这个技巧后的效果"（激发兴趣），再讲"怎么做到的"（传授方法），最后讲"为什么这样更好"（加深理解）。

5. **对比教学**：大量使用"新手常见写法 vs 推荐写法"的对比，差异越大越好，让读者"一看就知道区别"。

6. **全中文表达**：技术术语保留英文（如 Prompt、Token、Vibe Coding），解释和论述全部用中文。

7. **适度幽默**：延续现有系列风趣幽默的风格，但不要过度，保持专业感。

8. **篇幅控制**：每篇文章正文控制在 5000-10000 字之间（不含代码和 Prompt 示例）。

9. **标题规则**：标题简明扼要，不要用"Vibe Coding：用自然语言写代码"这种冒号分离格式，直接用简短描述即可。不要中英文混杂解释。

10. **减少AI味儿**：行文要自然，像真人在写博客。不要用"让我们"、"接下来"等过渡词，不要每段开头都是"首先…其次…最后…"。

## 关键原则

1. **每篇文章都要先读取规划文档** `vibe_coding_series_plan.md`，确认内容要点不遗漏。
2. **先读取一篇现有文章**（如 `src/go_series/go_advanced/channel.md`），确认格式一致。
3. **Prompt 示例必须真实可用**，不要编造看起来很炫但实际效果很差的 Prompt。
4. **图片占位标记不能少**，操作型文章至少 8 张，其他类型至少 5 张。
5. **不需要前后衔接**：开头不回顾上篇，结尾不预告下篇。每篇文章独立成文。
6. **结尾推广块必须加**，使用上面提供的统一 HTML 模板。
7. **面向中国用户**：推荐工具时优先考虑国内可用性，涉及付费工具时说明价格和免费替代方案。
8. **长文章主动拆分**：如果按规划文档写某篇文章时发现内容量过大（正文超过 10000 字），应主动提出拆分方案，经用户确认后再写。
